電話対応 / 0–15分

最初の15分は「聞く」が9割。判断は指揮へ渡す。参照: 01 受電ヒアリングシート
  • ✓
    受電したら時刻をメモ (mm/dd HH:MM)
    以降すべての記録に時刻を付ける。曖昧語(さっき・いま)は禁止。
  • ✓
    相手の名前・所属・連絡先・かけ直し先を確認
    「お名前とご所属、この後こちらからお電話できる番号を伺ってもよろしいですか?」
  • ✓
    「何が」「いつから」「どうなっているか」を聞く
    起こっている事象を相手の言葉のまま書く。解釈は後で。
  • ✓
    同じ現象を複数ユーザが見ているか確認
    1人だけ / 複数 / 全員 を区別。障害範囲の初期切り分け。
  • ✓
    相手の手元で再現できる操作を1つだけ聞く
    URL・画面・エラーメッセージ。こちらで同じ操作を再現する材料。
  • ✓
    「調査して折り返します」で一旦切る
    この場で原因や対処を断定しない。復旧時刻を約束しない。
  • ✓
    01 受電ヒアリングシートに書き起こす
    口頭メモをそのまま残さず、必ずシート化してチームに渡す。
  • ✓
    指揮係に声で「1件入った」と宣言する
    無言でシートを置かない。音で共有。
⚠ ユーザ申告と観測事実を分けて書く。「落ちている」は申告、「tcp/443応答なし」が事実。

指揮 / 0–15分

情報を集約し、優先度と役割分担を決める。参照: 02 引き継ぎ / 05 全体チェック / 02.1 先にやること
  • ✓
    受電内容をホワイトボード or 共有ドキュメントに書く
    個人メモで抱え込まない。時刻・事象・影響範囲の3点を先に。
  • ✓
    「事実」「申告」「推測」をラベル分け
    後から見返すときの誤判定を防ぐ。
  • ✓
    4役割を明示 (電話/指揮/サーバー/ネットワーク)
    誰が今どれをやっているか全員が言えるようにする。
  • ✓
    02 引き継ぎテンプレに記入して技術へ渡す
    口頭だけで渡さない。書いたものを渡す。
  • ✓
    02.1「先にやること」の順序を守らせる
    電話→再現→環境→構成→ログ→(最後に)攻撃種別判定。先に断定しない。
  • ✓
    対外連絡は指揮経由で一本化する宣言
    技術担当を直接顧客に出さない。
  • ✓
    同時多発を警戒 (陽動の可能性)
    1件目に集中している間に2件目が来る想定で電話応対係にヘルプを準備。
⚠ この段階で「ランサムです」「DDoSです」と断定しない。症状ベースで言う。

サーバー / 0–15分

状態の「観測」が先。止める判断は指揮。参照: 02.1 先にやること / 06 DDoS詳細
  • ✓
    対象サーバの FQDN と IP を確定
    観測点と被害サーバは別物の可能性。資産台帳で裏取り。
  • ✓
    疎通・応答を確認 ping curl -I
    応答コード・応答時間・ヘッダ。死活と遅延を切り分け。
  • ✓
    プロセスと負荷 top ps uptime
    CPU/メモリ/ロードを数値で記録。感覚語を使わない。
  • ✓
    接続状況 ss -ant | awk '{print $1}' | sort | uniq -c
    SYN-RECV / ESTAB の数を出す。閾値: SYN-RECV 100超、ESTAB 200超は異常寄り。
  • ✓
    ログの先頭1画面だけ見る (tail -n 100)
    最新のエラー/警告を2〜3行だけチーム共有。深追いは中間フェーズで。
  • ✓
    自分では再起動しない・設定を書き換えない
    証拠が消える。指揮の判断を待つ。
  • ✓
    OS/ミドルを確認 (RHEL/Debian/FreeBSD, apache/nginx)
    コマンド体系が違う。cat /etc/os-release / uname -a
⚠ 「落ちている」と言われても、実は DB 詰まり・外部 API 障害・DNS・証明書 の可能性が高い。DDoS 断定は最後。

ネットワーク / 0–15分

入口から観測。遮断は指揮の合図で。参照: 06 DDoS詳細
  • ✓
    経路を図にする (Internet → FW → LB → Srv)
    どこの数字を見ているのかを明示する。観測点のズレを防ぐ。
  • ✓
    外向け回線の使用率を確認
    平常比で何倍か。数字で残す。
  • ✓
    FW/ルータの conntrack/セッション数を確認
    上限に近いかを見る。YAMAHA は show status pp 系、Cisco は show conn count
  • ✓
    送信元の偏りを1行で
    特定の /24 や国が極端に多くないか。後で ACL 候補になる。
  • ✓
    DNS 応答を確認 dig @resolver FQDN
    名前解決か到達性かを切り分け。
  • ✓
    ACL/フィルタは「まだ入れない」
    YAMAHA ip pp secure filter は上書き事故が起きるので下書きだけ。
  • ✓
    AWS の場合: CloudWatch の NewConnections / Requests を見る
    SG/NACL を即更新しない。観測優先。
⚠ 遮断は強力だが正常ユーザも切る。遮断判断は必ず指揮の合図で。
