## 0. TL;DR (v3 — 20 行で全部)

- テスト環境は **bravo / victor の 2 台構成**、OS も役割も完全に別。
- **victor = 本番 Web + Mail + DHCP + SNMP + 内部ルータ的な "主役"** (Rocky Linux 8.10, PHP 7.2.24 EOL、古いOSSだらけ)。WordPress + Rainloop + 架空企業サイト (backup_html) を抱える。
- **bravo = DNS + 内部掲示板 + 学生用 sb_blog_* DB の "裏方"** (FreeBSD 14.3, 全ソフト最新)。
- **⚠️ この環境は過去のコンテスト (com1.local ドメイン = 1 班) の設定を使い回している** (§3 に物証)。
- 運営は **2026-03-09 に初期構築、2026-04-15 16:03 に仕上げ** (§4 dnf history 復元)。
- bravo のサーバ稼働期間 = **4 日 17 時間** = **2026-04-20 前後に起動** = 運営が言った「4/20-24 触れる期間」と完全一致 → demo 専用VM。
- 今年固有の追加は **SNMP/MRTG (4/15)** だけ = 監視系が当日シナリオの軸。
- victor に **rainloop 1.12.0 (2018 年版)** + **WordPress 4.9.4 (2018)** + **PHP 7.2.24 (2019, EOL)** + **telnet :23** + **HTTPS 無し** + **iptables 空** + **SELinux Disabled** + **pkexec SUID (PwnKit 対象)**。ほぼ全て既知 CVE 直撃。
- **bravo はソフトが最新でも pf/ipfw 無効 (= L3 防御ゼロ)、HTTPS 無し、IMAP 平文のみ、syslog 514/UDP 外部受信可、MySQL X Protocol 33060 外部公開**。
- 🔴 **bravo BIND の致命的設定: `allow-update { 10.0.0.0/8; }`** → 10.x 全域から `nsupdate` で **DNS レコード書き換え可能 = DNS spoofing / redirect 攻撃が現実的**。
- **新セグメント発見**: `10.1.129.0/24`(obuchi 接続元)、`10.1.130.0/24`(DNS forwarder 経由)。
- **新サーバ発見**: `10.1.6.254`(bravo の default GW = コア FW/Router と推定)、`10.1.130.1`(BIND forwarder + AXFR 許可先 = DNS secondary)。
- SNMP コミュニティは **`public` (read-only)** だが view は `.1.3.6.1.2.1.1` (system) + uptime のみに絞られてる (比較的まとも)。
- UDP :34020 は **dhcpd の内部用**、怪しくない。telnet :23 は **systemd telnet.socket** (per-connection 起動)。
- DNS ゾーン `com6.local` は bravo 権威。`www / mail → victor`、`ns → bravo`。AXFR は `10.1.130.1` のみ許可 (それ以外は拒否、まとも)。
- DHCP 配布レンジ `10.1.16.50-99`。**`not authoritative;`** なので偽 DHCP に押し負け得る。逆引き zone 名は **typo** (`in-add.arpa`)。
- 学生アカウント **100 人分**が全員 bash + Maildir 持ち。**`obuchi` ホームだけ 777 (踏み台候補)**。obuchi ユーザが **深夜 01:40-04:00 に 10.1.129.10 からログイン履歴あり** (運営テストか他チームか要確認)。
- **4/24 00:32 に victor がリブート済**。bravo は4/20頃から連続稼働。
- ⚠️ **bravo では manage が sudoers に入っていない** (victor とは違う)。4/24 の調査中に `sudo cat` を試みて失敗、認証ログに1行残った (§15 参照)。
- 当日確実に刺さる脆弱性 TOP3: **① rainloop 1.12.0 (CVE-2022-29360 等)、② WordPress 4.9.4、③ PHP 7.2.24 + 無制限設定**。
- 当日追加で覚えておく攻撃面: **④ BIND allow-update による DNS 書き換え、⑤ MySQL X 33060 外部公開、⑥ syslog 514 への偽ログ注入**。
- 🆕 (v3.1) **運営実図判明** = 2024 年予選環境の流用確定。10.1.6.254 は **RTX1200** (ヤマハルーター、L3のみ)、前段に **RTX1210 (VPN/ゲートウェイ)**、CIC 側に **Cisco ASA** (別系統 10.1.36.0/24)。**bravo/victor は ASA の保護下ではない** = FW 的な防御はルーター ACL 程度。
- 🆕 ネットワーク機器も攻撃対象: **RTX1210/1200 (ヤマハ)、C3560-CX (Cisco SW)、Buffalo AP、ASA** → 古いファーム / デフォルト認証情報 / Cisco SMI 等。
- 🆕 外部入口は **RTX1210 (IPsec/PPTP) の .151 (133.42.49.x)** が唯一。Internet → VPN → 10.1.16.0/24 → RTX1210(.254) → 10.1.26.0/24 → RTX1200 → 10.1.6.0/24 → bravo/victor の一本道。

#### named.conf 内容 (抜粋)
```conf

    directory       "/usr/local/etc/namedb/working";
    pid-file        "/var/run/named/pid";
    listen-on       { any; };
    forwarders      { 10.1.130.1; };             ← 外部問い合わせは .130.1 へ転送
    allow-query     { 10.0.0.0/8; };             ← 10.x 全域から問い合わせ可 (広め)
    allow-update    { 10.0.0.0/8; };             ← 🔴🔴🔴 10.x 全域から nsupdate 可!
    dnssec-validation no;
};

zone "." { type hint; file "named.root"; };

zone "com6.local" {
    type master;
    file "../primary/com6.local.zone";
    notify yes;
    allow-transfer { 10.1.130.1; };              ← AXFR は .130.1 のみ (厳しめで良い)
    also-notify   { 10.1.130.1; };
};
zone "6.1.10.in-addr.arpa"      { ... 同上 ... };
zone "service.com6.local"        { ... 同上 ... };
zone "130.1.10.in-addr.arpa" {                    ← 逆引き forward
    type forward;
    forward only;
    forwarders { 10.1.130.1; };
};
```

#### 🔴 致命的設定: `allow-update { 10.0.0.0/8; }`
- **10.x.x.x 全域のホストから `nsupdate` で DNS レコードを書き換えられる**
- 典型的な攻撃手順:
  ```bash
  nsupdate -v
  > server 10.1.6.1
  > zone com6.local
  > update delete www.com6.local. A
  > update add    www.com6.local. 300 A <攻撃者IP>
  > send
  ```
- これで **`http://www.com6.local/` (WordPress) を任意のサーバーに向けられる** = 本物のWPを見ているつもりが攻撃者サイトにリダイレクト
- **当日のシナリオとして極めて有力**

#### 管理ゾーン
- 正引き: `com6.local`, `service.com6.local`
- 逆引き: `6.1.10.in-addr.arpa`
- 逆引き forward: `130.1.10.in-addr.arpa` → `10.1.130.1`
- forwarders: `10.1.130.1` (外部問い合わせの転送先)

## 8. 脆弱性・違和感リスト

### 8.1 🎯 当日確実に刺さる攻撃面 TOP6 (v3)

#### ① Rainloop 1.12.0 (2018 年版) [victor]
- `http://www.com6.local/rain/` で到達
- 既知 CVE: **CVE-2022-29360 (任意ファイル読取)**, **CVE-2020-12641 (phpinfo経由情報漏洩)**, **CVE-2019-10025 (XSS)**
- **HTTPS 無し** でログイン画面 = **ID/PW 平文流出**
- `manage@com1.local` の設定が残留している = **前年の認証情報が手掛かりになる可能性**

#### ② WordPress 4.9.4 (2018-02) [victor]
- `http://www.com6.local/` (DocumentRoot) で到達
- 既知 CVE: **CVE-2018-6389** (wp-admin/load-scripts.php DoS), XML-RPC 総当り, REST API 経由の権限昇格
- **プラグインは全部デフォルト** (akismet, hello.php, wp-multibyte-patch)、攻撃面は WP 本体

#### ③ PHP 7.2.24 + 設定無制限 [victor]
- **CVE-2019-11043** (PHP-FPM RCE): nginx + 特定パスで刺さる。httpd 経由なら限定的だが要確認
- `allow_url_fopen=On` + `disable_functions=空` + `open_basedir=空` → **RCE 発火時にサーバ全体が取れる**

#### ④ 🆕 BIND `allow-update { 10.0.0.0/8; }` [bravo]
- `nsupdate -v` で 10.x から DNS レコードを書き換え可能
- 例: `www.com6.local` を偽IPに差し替え → WordPress 利用者が攻撃者サイトに誘導
- **当日のシナリオ最有力候補**: 「WP のログイン画面が偽物」「トップページが別ホストに向いている」等

#### ⑤ 🆕 MySQL X Protocol 33060 外部公開 [bravo]
- `mysqlsh --mysqlx -u root -h 10.1.6.1 --port=33060` 等で接続試行可
- MySQL 8 の管理プロトコル。認証突破できれば直接 SQL 実行
- 対策: 3306 のように localhost 限定にすべき (今は `*:33060`)

#### ⑥ 🆕 syslog 514/UDP 外部受信 [bravo]
- `logger -n 10.1.6.1 -P 514 ...` で偽ログ注入可
- インシデント調査時の誤誘導、SIEM への汚染など
- 運用上は「中央ログ集約サーバーの役割」の可能性もあるが、ACL 無しは危険

## 12. 本番直前メモ (team 向け / v3)

1. **書き換え・再起動は最後の最後まで保留**。共有環境なので一手で全チームに迷惑。

   - ② WordPress 4.9.4 改ざん (wp-login 総当り / REST API) (victor)
