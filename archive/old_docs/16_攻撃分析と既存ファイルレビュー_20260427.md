# Part 1. 検証環境ログから抽出した想定攻撃と対策

## 1.1 環境概要

- **bravo (10.1.6.1)**: FreeBSD 14.3 / Apache 2.4.66 / PHP 8.5.1 / MySQL 8.0.44 / BIND 9.20.20 / Sendmail 8.15.2 / courier-imap
- **victor (10.1.6.2)**: Rocky Linux 8.10 / Kernel 4.18.0-553 / Apache 2.4.37 / PHP 7.2.24 (EOL) / MariaDB 10.3.39 / Dovecot 2.3.16 / RainLoop 1.12.0 / WordPress 4.9.4
- ホスト名 `*.com6.local` のほか、設定中に `com1.local` の痕跡 (設定流用・他ドメインの IMAP 資格情報残置)
- 接続経路: クライアント → IPSec/PPP-VPN (133.42.49.151) → 10.1.6.0/24

## 1.2 最重要発見 (優先度: 最高)

**A. 既に侵害されている可能性が高い痕跡** (log3 の `last` 出力)
2026-04-24 01:40〜04:00 に `obuchi` および `manage` が **10.1.129.10** から長時間ログイン。深夜帯・通常業務外・想定外セグメントからのアクセスで、すでに踏み込まれている前提で対応するのが妥当。

**B. DocumentRoot 配下に MySQL 認証ファイル**
`/var/www/cgi-bin/.my.cnf.6804` と `.mysql.6804` が Web 公開ツリーにあり、URL 直叩きで DB 認証情報が取得可能。事実上「どうぞ侵入してください」状態。

**C. /home/obuchi が drwxrwxrwx (777)**
誰でも `~/.ssh/authorized_keys` を書けるため、低権限ユーザを取れば即 obuchi に化けられる。

## 1.3 想定攻撃シナリオ (キルチェーン)

### シナリオ① 外部偵察 → Web 経由 RCE → 横展開 → DB 全件持ち出し

1. **偵察**: SNMP `public` コミュニティ (net-snmp 5.8) で MIB 列挙 → OS / プロセス / インターフェース情報を取得
2. **初期侵入**: WordPress 4.9.4 または RainLoop 1.12.0 (CVE-2022-29360 等) の既知脆弱性で RCE
3. **権限昇格の準備**: PHP 7.2.24 (EOL) + `disable_functions=空` + `open_basedir=空` + `allow_url_fopen=On` → `system()` で任意コマンド実行
4. **認証情報奪取**: `/var/www/cgi-bin/.my.cnf.6804` から MySQL クレデンシャル → MariaDB (`*:3306` 全公開) へ直接接続
5. **永続化**: SELinux Disabled + firewalld inactive のため、cron / at / SUID で自由に居座り
6. **横展開**: `manage` アカウント (両機 wheel/sudoers、ログインソース `10.1.16.x` 多数) を奪取し bravo へ。`toor:0` で監査回避
7. **データ持ち出し**: dovecot `passdb=pam` のため shell パスワード = メールパスワード。IMAP 143 (平文) で全社員のメール窃取

### シナリオ② 内部 MITM → DNS 書換 → フィッシング

1. DHCP `not authoritative` + DDNS 鍵なし → rogue DHCP で別 DNS を配布
2. BIND の `allow-update {10.0.0.0/8;}` + `dnssec-validation no` → `nsupdate` で MX/A レコードを書換
3. SMTP/IMAP 平文認証を傍受 (Telnet 23 も生きている)
4. なりすましメール送信: SPF/DKIM/DMARC 未設定

### シナリオ③ VPN→上流信頼を悪用した横断

1. IKEv1 (racoon, ISAKMP UDP/500・4500) で aggressive mode が許容されていれば PSK クラック
2. sendmail.cf や RainLoop データに残る `com1.local` ドメインへ設定信頼を辿って侵入伝播
3. forwarders `10.1.130.1` への信頼を悪用して上流 DNS も汚染

## 1.5 即時実施すべき封じ込め (優先順)

### 短期 (本日中)

1. `/var/www/cgi-bin/.my.cnf.6804` と `.mysql.6804` を**即削除**
2. `/home/obuchi` を 700 に変更し `~/.ssh/authorized_keys` を確認
3. `obuchi` `manage` のパスワードを全機リセット、SSH 鍵を再発行
4. `telnet.socket` 停止、IMAP 143 / MariaDB 3306 / MySQL X 33060 を bind 制限
5. firewalld / pf を有効化し、必要ポートのみ許可

### 中期 (数日)

6. SELinux を enforcing 化
7. WordPress / RainLoop / PHP / BIND / squid を最新化または撤去
8. Apache `<Directory />` を最小権限化、`.htaccess` 不可
9. sendmail を再生成、SMTP AUTH+STARTTLS、SPF/DKIM/DMARC 投入
10. BIND の `allow-update` を TSIG 必須化、`dnssec-validation auto`

### 長期 (継続運用)

11. manage 共有アカウントを廃止、Bastion + MFA
12. IKEv2 + 証明書認証へ移行
13. 集中ログ基盤 (TLS syslog 6514) と auditd 整備

> **最も危険な単一発見**は「DocumentRoot 配下の `.my.cnf.6804`」と「/home/obuchi の 777」。この 2 つだけで認証なしの DB 全件取得 + obuchi 完全乗っ取りが成立する。深夜の不審ログイン痕跡と合わせ、すでに侵害されている前提で対応するのが安全。

## 2.4 既存ファイルにも無い・薄い観点 (私の分析が補える部分)

- **`toor:0` UID0 重複アカウント** (bravo /etc/passwd)

- **at/atd 永続化リスク**、**git_daemon ユーザ**
- 各弱点に対する**具体的な設定変更例** (disable_functions の値、ssl_min_protocol 等)

# Part 3. 既存 14/15 番ファイルのレビュー

## 3.1 14 番 (調査レポート v3.1) のレビュー

**総合評価: ★★★★☆ (4.5/5)**

### 強み

1. **コンテキスト把握が非常に強い**: dnf history から運営の作業タイムラインを逆算、SCCS 2024 流用の物証連鎖を組み立てている。単なる「弱点リスト」を超えて「運営の意図」まで踏み込んでいる
2. **TOP6 攻撃面**の優先度付けが現場運用に向く粒度
3. **§9 シナリオ→対応サーバ即決マップ**は当日の通報対応で即引ける形になっており、実用性が極めて高い
4. **§10「まだ確認していない事項」**を残しているのが正直で良い (知らないことを知らないと書く規律)
5. **read-only の倫理規範**が明示されている
6. **共有環境であることへの配慮**が随所に入っている

### 弱点・改善提案

| # | 指摘 | 提案 |
|---|---|---|
| 1 | **「対策 (緩和策)」が無い** — 攻撃面の列挙はあるが「もし当日この攻撃が来たらどう封じ込めるか」の具体手順が§9 の調査コマンドどまり | TOP6 各々に「初動対応 (read-only 範囲) + 運営合意後の遮断手順」を追記 |
| 2 | **dovecot `passdb=pam` の影響度が過小評価** — §6.9 に書かれているが、100 学生全員が IMAP 経路で認証試行できる構造は単独で TOP 級。TOP6 に入っていない | TOP7 に格上げ、または PHP 7.2.24 と並列に |

| 4 | **`/var/www/cgi-bin/.my.cnf.6804`** は「最高優先で中身確認すべき」項目だが「未確認 (§10.1)」止まり | 644 で読めるかだけでも `ls -l` で確認しておく (read-only)。ファイル名が `.6804` の意味 (uid? PID? バックアップ?) も推測しておく価値あり |

| 11 | **§7 比較表に「攻撃連鎖図」が無い** — 個別弱点はあるが、「victor RCE → manage 奪取 → bravo へ ssh → DNS 書換 → 全社員フィッシング」のような**キルチェーン視覚化**がない | 1 ページの flow 図を§8.1 直後に追加 |
| 12 | **ベースライン情報 (現在のプロセス・接続・ユーザ) が日付付きで残っていない** | 当日の比較に使う「2026-04-24 12:15 時点の正常スナップショット」として ss の出力等を Appendix にダンプしておけば、当日の異常検知が速くなる |
| 13 | **§13 過去コンテストキーワード**が良い導線だが、**実際の過去 wrap-up へのリンク**は無い | 同じディレクトリに過去 wrap-up を保存しておくと当日参照が速い |
| 14 | **`/etc/ssh/sshd_config`** が§10.1 の未確認に入っているが、これは PubKey 強制の有無・PermitRootLogin で当日の侵入経路判定に直結 | 644 で読めるはずなので事前読了推奨 |
| 15 | **記述が長く、当日 5 秒で参照する項目が埋もれる** | 1 ページ「当日カンペ」を別ファイルに切り出す (IP・パス・コマンド・通報→サーバ判定だけ) |

### 倫理・コンプラ観点

- **「事前ログイン禁止」のルールがある可能性** に言及はあるが、確認手段が「もしあるなら」止まり。可能なら**運営に事前確認**しておくのが完全に安全 (してあるなら本書に明記)
- **共有環境での read-only 徹底**は素晴らしいが、「他チームの活動を傍受しない」 (他ユーザの操作中ファイル `lsof` 等を覗かない) 宣言があるとさらに守備が固い
- 認証情報 (`sh1Ra8mA`) が 14 番に平文で書かれているので、**この 2 ファイルを共有する範囲・保存場所**もルール化したい

## 3.3 2 ファイルを通したメタ提案

1. **3 つ目のファイル「対策・初動プレイブック」を作る** — 14 番に弱点があり、15 番に作業方針がある。**「もしこの攻撃が来たら、read-only でこう調べ、運営合意後にこう封じる」**を 1 枚にまとめると現場で強い

3. **過去コンテスト wrap-up を `/参考資料/`** に保存 (SCCS 2018-2024)。当日は外部検索が遅い/できない可能性
4. **「やらないこと」ホワイトリスト** が今は §6.1 にあるが、もう一段絞って「本当に絶対に押すな」 5 個 (`>`/`>>`/`systemctl restart`/`vi`/`nsupdate`) に縮約したシンボル付き図を作ると徹底しやすい
5. **チーム内 dry-run** を本番前にもう 1 回。「通報 → 14 番§9 引く → 15 番ルール確認 → 報告書テンプレ」までの所要時間を測る
