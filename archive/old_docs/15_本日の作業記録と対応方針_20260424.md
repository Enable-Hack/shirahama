  - ファイルの書き込み・改変は一切していない（read のみ）
- 本件は **FreeBSD の sudoers 設計の違い** に起因（manage は `wheel` グループだが sudoers 明示登録なし）

**設定変更・サービス制御・ユーザー管理に sudo は一切使用していない**（両サーバー共通）。

## 5. 管理者から聞かれた場合の対応（模範回答）

### 5.1 基本スタンス

> **正直に話す。隠さない。嘘をつかない。**
>
> 事実として「破壊も改変もしていない」ので、堂々と説明できる。変に隠すと逆に疑われる。

### 5.2 想定 Q&A

**Q1: 「manage ユーザーで何をしていましたか？」**
> A: 「情報危機管理コンテストに向けた事前の環境調査です。与えられた demo 環境がどういう構成か把握するために、OS バージョン・稼働サービス・主要な設定ファイル・ネットワーク構成を閲覧していました。**設定変更やサービス操作は一切していません**。」

**Q2: 「具体的にどんなコマンドを？」**
> A: 「`uname`, `ps`, `netstat`, `ifconfig`, `cat`, `ls`, `find`, `rpm -qa`, `php -i`, `dnf history`, `last` などの **読み取り系コマンド** です。`/etc/` 配下の設定ファイルを `cat` で閲覧しました。**書き換え系（`vi`, `sed`, `echo >`）や制御系（`systemctl`, `service`）は使用していません**。」

**Q3: 「`/tmp/` にファイルを作っていましたか？」**
> A: 「はい、`find` や `rpm -qa` の出力を後で参照できるように一時的に `/tmp/` にリダイレクトしていました。調査が終わった後、**全て `rm` で削除済みです**（`ls /tmp/` で確認可能）。中身は `find` コマンドの出力そのものなので、システムには何の影響もありません。」

**Q4: 「なぜ `sudo` を使いましたか？」**
> A: 「`find` を `/` 全体に流した時、manage 権限で読めないディレクトリの権限エラーを抑えるためと、パーミッション制限のある `/etc/dhcp/dhcpd.conf` 等の **閲覧目的** のみです。**設定変更目的の sudo は使用していません**。`/var/log/secure` や `.bash_history` で確認可能です。」

**Q5: 「これはルール違反ですか？」**
> A: 「manage ユーザーは運営から提供された正規の認証情報で、事前の環境把握は本番コンテストに向けた準備として一般的な行為と認識しています。もし運用ルールで『事前ログイン禁止』や『特定コマンドの禁止』がある場合は、私の認識漏れですので、以後注意します。**ログは全て残っているので確認いただけます**。」

**Q6: 「他のサーバーにも入りましたか？」**
> A: 「bravo (10.1.6.1) と victor (10.1.16.1 / 10.1.6.2) の 2 台のみです。他のサーバーへは入っていません。」

**Q7: 「他のユーザーのファイルを見ましたか？」**
> A: 「`/etc/passwd` などシステム共通の設定ファイルは閲覧しましたが、**個別ユーザーのホームディレクトリの内部は閲覧していません**。Rainloop の data ディレクトリは Web アプリの共有データなので、そこに含まれる旧設定ファイルは構成調査の一環として確認しました。」

**Q8: 「bravo で `sudo` が拒否されたログが残っていますが、これは何ですか？」**
> A: 「はい、`sudo cat /usr/local/etc/namedb/named.conf` を試した際に拒否されました。FreeBSD の manage ユーザーは wheel グループには入っていましたが sudoers ファイルに明示登録されていなかったため、sudo 自体が使えないことをその時初めて知りました。拒否された後は **一般権限で同ファイルを読み取りました**（そもそも 644 パーミッションで誰でも読める設定でした）。書き込み・改変は一切していません。これは sudoers の設定理解不足による一度のミスで、悪意はありません。」

**Q9: 「BIND の設定ファイルを読んだ目的は？」**
> A: 「稼働サービスの構成把握の一環です。稼働している BIND のゾーン設定・forwarder 設定・DNSSEC の状態を確認しました。その過程で `allow-update { 10.0.0.0/8; }` という重要な設定を確認しましたが、これは**読み取っただけ**で、`nsupdate` 等によるゾーン書き換えは一切していません。本番コンテストでは、こういう設定があった場合に備えて対処方法を準備しておく、という学習目的です。」

**Q10: 「/etc/passwd や /etc/rc.conf などの設定ファイルは書き換えていませんか？」**
> A: 「**mtime / ctime を確認いただければ**、本日の調査時刻より前のままであることが分かります。`stat /etc/rc.conf` 等で ctime が今日の日付になっていないことを確認可能です。閲覧（atime 更新）のみで、変更は一切していません。」

### 5.5 無実を証明できるログ（管理者に見てもらえばよい）

**victor (Rocky Linux)**
```
/var/log/secure       → SSH 認証履歴（正規ログインの証明）
/var/log/wtmp         → ログイン履歴（last コマンドで表示可能）
/var/log/audit/audit.log → auditd が有効なら全 syscall が記録
~/.bash_history       → 実行コマンド履歴（読み取り系のみの証明）
/etc/*, /var/www/*    → mtime/ctime が本日の調査時刻より前のまま（改変していない証明）
```

**bravo (FreeBSD)**
```
/var/log/auth.log     → sshd / sudo 履歴（sudo 1回拒否が明示的に記録されている）
/var/log/messages     → syslog 全般
/var/log/wtmp         → last で表示可能
/var/log/utx.log      → FreeBSD のログイン履歴
~/.history (manage)   → csh 実行履歴（改竄無し）
/etc/*, /usr/local/etc/* → mtime/ctime 未変更
```

→ **このログ群が無実を証明する最大の武器**
→ 特に FreeBSD の `/var/log/auth.log` に記録されている sudo 拒否 1 件は「やろうとしたが出来なかった」証拠として逆に無害性を示す

## 6. 今後のルール（作業者・アシスタント共通）

### 6.1 書き込み系コマンドの禁止
- ❌ `>`, `>>`, `tee`（ファイル書き出し）
- ❌ `mkdir`, `touch`, `cp`, `mv`
- ❌ `rm`（調査痕跡除去を除く）
- ❌ `vi`, `sed -i`, `echo >>`（編集）
- ❌ `systemctl start/stop/restart` (Linux)
- ❌ `service ... start/stop/restart` (FreeBSD)
- ❌ `dnf install`, `pkg install`, `pkg upgrade`
- ❌ `nsupdate`（DNS 書き換え）
- ❌ `pfctl -e/-d`, `ipfw`（FW 変更）

### 6.2 読み取り系のみ推奨
- ✅ `cat`, `less`, `head`, `tail`, `grep`
- ✅ `ls`, `find`（`-exec rm` 無し）
- ✅ `ps`, `netstat`, `ss`, `sockstat`, `ifconfig`
- ✅ `ssh`, `scp`（受信のみ）
- ✅ `systemctl status`, `systemctl list-units`（Linux）
- ✅ `service -e`, `service <name> status`（FreeBSD）
- ✅ `pkg info`, `rpm -qa`, `dnf list installed`（参照のみ）
- ✅ `dig`, `nslookup`（ゾーン転送は運営確認後）

### 6.3 DB 操作
- ✅ `SELECT` のみ
- ❌ `INSERT`, `UPDATE`, `DELETE`, `DROP`, `CREATE`, `ALTER`

### 6.4 出力管理
- 画面表示のみ（`| less`, `| head`）
- メモは **ローカル PC 側** に取る
- サーバー側のファイル生成は原則禁止

### 6.5 sudo / 権限昇格の取り扱い（今日の教訓）
- sudo を使う前に **`sudo -l`** で自分の sudoers 設定を確認する
- 拒否された場合は **再試行しない**（ログが溜まるだけ）
- FreeBSD では wheel グループ ≠ sudoers、`su` は別途パスワードが必要な点に注意
- sudo が使えない状況でも、ファイルのパーミッションを `ls -l` で確認すれば一般権限で読めることが多い
