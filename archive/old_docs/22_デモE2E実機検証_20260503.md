# 22. デモ環境 4 層パイプライン E2E 実機検証 (2026-05-03)

## 概要

OCI デモ環境 (attack-vm + bravo + victor) で AI ログ分析パイプラインを **手動で 4 層完走** させた記録。観測層 → 判断層 1 (Mock) → 判断層 2 (Claude) → 検証層 が、実機ログ・実 API で動作することを確認。

検証シナリオは **wp-tamper**（WordPress 認証ブルートフォース + 過去テストの攻撃痕跡が混在）。


## つまずいたポイント（教訓）

| 症状 | 原因 | 対処 |
|---|---|---|
| Claude Code が `ssh manage@10.1.1.2` を提案 | `incident.md §1` が本番 IP ハードコード | デモでは「No」を押して `ssh victor` で手動実行（または環境変数化が恒久対応） |
| `python3 -c "..."` の複数行ペーストで f-string 壊れた | ターミナルが途中で別コマンド受信 | `cat > /tmp/x.py <<'EOF' ... EOF` で heredoc 化してから実行 |
| `python3 /tmp/run_mock.py` で `ModuleNotFoundError: agent` | スクリプトが `/tmp` にあるので sys.path に shirahama が入らない | `PYTHONPATH=. python3 /tmp/run_mock.py` で cwd を追加 |
| `from anthropic import Anthropic` で ModuleNotFoundError | anthropic SDK 未インストール | `pip3 install anthropic` |
| `analyzer.run('/tmp')` が 0 件 | nginx 用 JSONL のファイル名が `access.log` 期待 | parse_clf.py の出力先を `/tmp/access.log` にする |
| permission prompt が英語コマンドだけで判断しにくい | Claude Code のデフォルト UI 仕様 | TODO: hook で日本語説明を挟む（後述） |

## 確認できたこと

- 観測層の正規表現が **本番想定（FreeBSD）でなく Rocky Linux でも動く**（access_log フォーマットは共通）
- Mock backend の whitelist が attack-vm IP を **誤って drop しない**（10.1.11.50-99 範囲外なので）
- Claude backend が **同じ攻撃に対し Mock と異なる書き方の patch を出す**（regex ではなく contains で書く傾向）
- Claude の `summary_ja` が **そのまま PukiWiki 完了報告に使える品質**

## 確認できていないこと

- 検証層 (`validator.py`) で実際に弾かれるケース（今回は 22/22 通過、validation 失敗パターン未確認）
- 提案層 (`renderers/nginx_renderer.py`) で nginx ルール文字列に変換するところ（未実行）
- 時間窓 jq フィルタの動作（今回は 82 行全部食わせた）
- DNS / mail / privesc 系シナリオ（bravo の BIND/Sendmail 未構築）
- 本番モード settings.json での挙動（build mode で動作確認）
