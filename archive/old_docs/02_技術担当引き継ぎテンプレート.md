# 02. 技術担当引き継ぎテンプレート

> 受電ヒアリング（01）で得た情報を整理し、**サーバ管理者／ネットワーク管理者** に渡すためのテンプレート。
> チーム内ホワイトボード、口頭ブリーフ、チャット共有のいずれでも使える形式。

---

## 使い方

1. 電話応対係が [01_受電ヒアリングシート](./01_受電ヒアリングシート.md) を記入
2. 記録係が本テンプレートに転記・整理
3. 指揮係が「技術調査に何を依頼するか」を明記して技術係に渡す
4. 技術係は調査完了後、**調査結果欄** を埋め返す

---

## 引き継ぎサマリ（ヘッダ：30秒で把握できる情報）

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
事案ID：          （例：INC-20260420-001）
受電時刻：        mm/dd HH:MM
報告者：          （所属／氏名／連絡先）
発生時刻(申告)：  mm/dd HH:MM
対象：            （サービス名／端末名／セグメント）
想定影響：        □業務停止 □情報漏えい疑 □外部からの攻撃 □未分類
緊急度：          □高（業務停止・拡大中） □中 □低
優先度：          □P1 □P2 □P3
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 緊急度の判定基準（迷ったら上位）

| 緊急度 | 判定基準 |
|---|---|
| **高** | 業務停止中／複数ユーザに拡大／外部に漏えいの可能性／顧客経営層から苦情 |
| **中** | 単一ユーザ／局所的障害／拡大は未観測 |
| **低** | 影響は確認されず要調査のみ／誤検知の可能性あり |

---

## 事実と申告の分離（**最重要**）

> 技術担当が誤った前提で動かないように、**事実** と **相手の申告** を必ず分けて書く。

```
【確認済み事実】
- 10:20 ○○ログに××のエントリあり（ログ取得時刻：10:25）
- 10:22 監視画面で CPU 90% を観測

【相手の申告（未確認）】
- 「メールの添付ファイルを開いたら画面が赤くなった」← ユーザ本人の申告のみ、現物未確認
- 「他の部署でも同じ症状」← 伝聞、裏取り未

【不明・未取得】
- 当該端末のホスト名・IP
- 添付ファイルの現物
- 症状の画面キャプチャ
```

## ネットワーク管理者向け：引き継ぎ項目

### N-1. 必ず伝える

| # | 項目 | 例 |
|---|---|---|
| 1 | 対象セグメント／VLAN | 営業部VLAN (VLAN10) |
| 2 | 対象IPアドレス／MAC | 192.168.10.45 / 00-11-22-... |
| 3 | 接続スイッチ／ポート | C3560-1 / Gi0/12 |
| 4 | 観測されている通信異常 | 外向き通信量 通常の50倍 |
| 5 | 発生時刻と現在までの推移 | 10:15 発生、10:30時点で継続 |
| 6 | 既に実施した操作 | なし／ケーブル抜線済み／VLAN変更済み |

### N-2. 依頼事項（指揮係が選択）

```
[ ] 当該ポート／VLAN のトラフィックを確認してほしい
[ ] 外向き通信（特に未知の宛先IP・不審なDNS）を調査
[ ] 該当端末を隔離VLAN に移動（※指揮承認後）
[ ] FW/UTM ログで当該IPの通信履歴を抽出（過去24h）
[ ] 他の端末で類似通信がないか確認
[ ] VPN接続中の外部ユーザに異常がないか確認
```

### N-3. 調査結果記入欄（NW管理者が埋める）

```
調査開始：mm/dd HH:MM   調査者：
[確認事項]
- 対象IPの通信先：
- 異常な通信の有無：
- 同様通信の他端末：
- FW/UTMログ特記事項：

[判断]
□ 封じ込めを推奨（隔離VLAN移動／ACL追加）
□ 経過観察で十分
□ 他セグメントへの拡大あり → 指揮へエスカレ

[提案アクション]
1.
2.

調査完了：mm/dd HH:MM
```

## サーバ管理者向け：引き継ぎ項目

### S-1. 必ず伝える

| # | 項目 | 例 |
|---|---|---|
| 1 | 対象ホスト名 | web01.example.local |
| 2 | OS／ミドルウェア | CentOS 7 / Apache 2.4 / WordPress 5.x |
| 3 | 稼働サービスと役割 | 社内ポータル（全社員利用） |
| 4 | 現在の状態 | 稼働中／応答なし／一部機能停止 |
| 5 | 観測された症状 | 500エラー多発／ログイン不能／改ざん疑い |
| 6 | 最後の正常動作時刻 | 10:00 時点ではアクセス正常 |
| 7 | 直近の変更 | 昨日パッチ適用／なし |

### S-2. 依頼事項（指揮係が選択）

```
[ ] ログ確認（/var/log/secure, /var/log/messages, アプリログ）
[ ] プロセス確認（不審な外部通信プロセス）
[ ] 認証ログで異常ログイン有無
[ ] Webアクセスログで不審リクエスト（過去24h）
[ ] ファイル改ざん確認（Webルート／設定ファイル）
[ ] バックアップからの復元が可能か確認（※復元実施は指揮承認後）
[ ] サービス停止／再起動の要否判断（※実施は指揮承認後）
```

### S-3. 調査結果記入欄（サーバ管理者が埋める）

```
調査開始：mm/dd HH:MM   調査者：
[確認事項]
- ログ特記事項：
- 不審プロセス：
- 認証ログ異常：
- 改ざん痕跡：
- 直近変更点：

[判断]
□ サービス継続可（監視強化）
□ 一時停止を推奨（影響拡大防止）
□ 侵害確定（復旧作業フェーズへ）

[提案アクション]
1.
2.

調査完了：mm/dd HH:MM
```

## 指揮・意思決定係向け：判断メモ

> 引き継ぎ後、**次の一手を決める** ための判断材料集約欄。

```
【NWからの報告】
【サーバからの報告】
【顧客側の状況】（顧客担当者からの追加情報があれば）

【判断事項】
1. 封じ込め実施の要否：□実施 □保留（理由：      ）
2. 顧客への中間報告：  □実施（時刻：   ） □次回観測後
3. サービス停止：      □実施 □継続 □一部停止（範囲：   ）
4. 追加調査依頼：      □NW側 □サーバ側 □両方

【次回チーム内レビュー時刻】
```

## チャット／口頭共有用の短縮フォーマット

> ホワイトボードや Slack 風の短文で共有する場合。

```
🔴[INC-20260420-001] 受電 10:20
対象: 社内ポータル (web01)
症状: 500エラー多発・ログイン不能（ユーザ複数名）
影響: 全社員 / 業務停止
状態: NW=調査中, サーバ=調査中
判断依頼: 10:40 まで
```

状態は随時上書き：

```
🟡[INC-20260420-001] 10:35
NW: 外向き通信 異常なし
サーバ: access_log に怪しいPOST多数（/wp-json/…） → 深掘り中
判断: サービス停止は保留、10:50 に再評価
```

---

## 引き継ぎ時の注意事項

- **時刻は必ず入れる**：「さっき」「いま」は禁止、mm/dd HH:MM で記録
- **断定しない**：「ランサム感染」ではなく「ランサム疑いの症状」
- **要求は具体的に**：「調べて」ではなく「web01 の access_log で /wp-json/ へのPOSTを抽出」
- **完了条件を添える**：「10分以内に一次判定」など時間と品質の目安を示す
- **連絡線を1本化**：技術担当から顧客へ直接返答させない（混乱防止）
- **記録は1枚にまとめない**：時系列が追えるよう **ヘッダ＋追記** 形式で更新する
