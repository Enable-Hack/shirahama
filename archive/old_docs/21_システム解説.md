## §1. 30 秒版（エレベーターピッチ）

> 「**電話受電起点のインシデント対応を AI で半自動化するパイプライン**です。Claude Code の slash command として `/incident` を叩くと、ログ取得から攻撃検知、顧客向け説明文の生成、次にやるべき深掘り skill の提示までを一気通貫でやる。**人間が判断と最終承認だけに集中**できる構成です。**破壊的なコマンドは settings.json で物理的にブロック**してあるので、AI が暴走して『触らない哲学』を破ることはありません」

**要点 3 つ**:
1. 電話受電 → 完了報告まで一気通貫
2. 人間判断は残す（リーダー Go/NoGo）
3. 物理ゲートで暴走防止

## §2. 5 分版（チームブリーフィング・運営説明）

### 2.1 解決する課題

白浜の競技は **電話受電から 30 分以内に技術対応 + 完了報告まで** やる必要があり、5 人体制でもインシデントが立て込むと捌けない。本システムは **観測・判断・検証の 3 層を AI で自動化**して、人間は **意思決定** と **顧客対応の温度感調整** だけに集中する設計。

### 2.2 4 層アーキテクチャ

```
電話受電
  ↓
[受電シート記入]
  ↓
[1] 観測層 (analyzer.py)        ← 41 pattern でログから攻撃を検出
  ↓ Signal[]
[2] 判断層 (Mock + Claude)      ← Mock で whitelist 除外 + 既知攻撃即 block
  ↓                              grey signal だけ Claude API で意味推定
[3] 検証層 (validator.py)       ← ReDoS / 壊れた regex を物理棄却
  ↓ PatchProposal[]
[4] 提案層 + 安全ゲート          ← settings.json deny/ask で破壊的コマンド遮断
  ↓
[人間判断: リーダー Go/NoGo]
  ↓
深掘り skill (/wp-tamper 等)
  ↓
PukiWiki に完了報告投稿
```

### 2.3 6 つの slash command

`/incident` がハブで、5 つの深掘り skill (`/wp-tamper` `/dns-tamper` `/ddos` `/phishing` `/ransomware`) がスポーク。受電したら最初に `/incident` を叩く。Claude が pattern_tag からカテゴリ判定して『次は /wp-tamper』のように提示するので、それを叩いて深掘りに入る。

### 2.4 「触らない哲学」の物理化

白浜は共有 demo 環境なので、**運営の出題前提を破壊するコマンド**（`nsupdate` でゾーン書き換え、`systemctl stop httpd` でサービス停止 等）を絶対に実行してはいけない。これを **思想の問題ではなく** `.claude/settings.json` の **`deny` ルールで物理ブロック**している。Claude が善意でそういうコマンドを提示しても、`Bash(*nsupdate*)` が deny ヒットして実行されない。

### 2.5 評価軸との対応

| 白浜評価軸 | 本システムの貢献 |
|---|---|
| 技術対応力 | 41 pattern で 6 カテゴリの攻撃を検知 |
| 顧客対応 | Claude が「現時点で確認できる範囲では」を含む 200字を自動生成 |
| 報告・連絡 | 経営層 300字 / 全社注意喚起 を同時生成 |
| プロセス遵守 | settings.json + Mock whitelist で「触らない哲学」を物理化 |

## §3. 30 分版（技術解説）

### 3.1 背景と動機（3 分）

私たちは **セキュリティ・キャンプ 2026 ミニ B トラック** で『AI ログ分析パイプライン』の 4 層アーキテクチャを学んだ。これは元々 Web の WAF ルール自動生成のための教材だが、**白浜の評価軸全部に応用できる思想**だと判断し、白浜用に拡張した。

白浜の特徴:
- **電話受電が起点** で、時間と顧客対応の温度感が採点に直結
- **複数攻撃カテゴリ**（Web / DNS / DDoS / フィッシング / ランサム）が並走
- **共有 demo 環境**で他チームに影響する操作は一発失格
- **30 分以内 / インシデント** が現実的目標

5 人体制でこれを捌くには、**人間判断を残しつつ、機械的部分は AI に任せる** のが現実解。

### 3.2 全体アーキテクチャ（5 分）

キャンプで学んだ **4 層分離** をベースにしている:

| 層 | 担当ファイル | 責務 |
|---|---|---|
| 観測層 | `agent/analyzer.py` | ログ → Signal[]、ルールベース、LLM 不要、可用性の下支え |
| 判断層 1 | `agent/backends/mock_backend.py` | 決定論的、whitelist filter + 既知パターン即提案 |
| 判断層 2 | `agent/backends/claude_backend.py` | LLM、grey signal の意味推定 + 顧客向け文面生成 |
| 検証層 | `agent/validator.py` | LLM 出力の壊れた提案を物理棄却 |
| 提案層 | `.claude/commands/*.md` | Claude Code が直接コマンド実行 |
| 安全ゲート | `.claude/settings.json` | 破壊的コマンドを deny / ask で物理ブロック |

**4 層を分離した理由は責任分界**:
- 観測層は『動き続ける』が責務
- 判断層は『質を上げる』が責務
- 検証層は『暴走を止める』が責務

LLM が不安定でも、**観測層と検証層は決定論的なので最低限の品質が保証される**。

### 3.3 構成要素 8 つを順番に（10 分）

#### ① analyzer.py — 観測層（41 pattern / 6 カテゴリ）

過去環境固有の CVE 名（rainloop 1.12.0 等）に過学習せず、汎用の攻撃カテゴリを捕捉する設計。

| カテゴリ | pattern_tag 例 | 入力ログ |
|---|---|---|
| Web app | `sqli/*` `xss/*` `path_traversal/*` `webapp/dotfile-access` `webapp/upload-php` `webapp/auth-bruteforce` `webapp/scanner-ua` `webapp/xmlrpc` `webapp/author-scan` | httpd_access_log |
| **CMDI** | `system-call` `shell-exec-call` `proc-open-call` `passthru-call` `popen-call` `exec-call` `backtick-exec` | httpd_access_log（URL 二重デコード対応） |
| DNS | `dns/unauthorized-update` `dns/axfr-attempt` `dns/amplification-bait` `dns/amplification-ratio` | named.log |
| Plain-text & Privesc | `protocol/telnet-access` `auth/ssh-failed` `auth/ssh-bruteforce` `privesc/pkexec-attempt` `privesc/sudo-unauthorized` `persist/at-job` | secure / auth.log |
| Mail | `mail/spf-fail` `mail/dkim-fail` `mail/relay-attempt` `mail/sasl-failed` `mail/burst` | maillog |

線形パターンマッチに加え、**集約検出**（同一 IP からの大量アクセス、ANY クエリ比率）も実装。

#### ② Mock backend — 判断層 1

LLM 不要の決定論的判断器。3 つの仕事:

1. **whitelist filter** — 配布アカウント（manage / root / admin）と自チーム IP（10.1.11.50-99）を drop。Claude API への流量が ~80% 減る見込み
2. **known-bad 即 block** — sqli / xss / cmdi / path_traversal / idor は専用ハンドラで即パッチ提案
3. **重複抑止** — rule_id 内容ハッシュで同一提案を advisory に降格

PDF「配布アカウントを使った攻撃は来ない」を物理化したのが whitelist。

#### ③ Claude backend — 判断層 2

grey signal（Mock のハンドラに無い種別）だけを Claude API に流し、意味推定 + 集約 + 顧客向け 200字 / 経営層 300字 / 全社注意喚起 を生成。**API コストとレイテンシを最小化**するため Mock の後段に置く。

#### ④ validator.py — 検証層

**LLM 出力を renderer に直接渡すな** がキャンプの核心メッセージ。7 種の検証で 1 件でも壊れていたら all-or-nothing で全棄却:

- rule_id の形式（注入対策）
- action の allowlist 適合
- target / match_value / rationale の空文字禁止
- **regex の ReDoS 検出**（catastrophic backtracking で WAF 自身が停止する事故防止）
- regex のコンパイル可能性
- confidence の範囲 [0.0, 1.0]
- match_value の長さ上限 512 文字

#### ⑤ /incident — エントリ slash command

**Claude 自身への指示書**。`/incident 13:00-13:30 victor` と打つと Claude が以下の 6 ステップを自動実行:

1. ssh でログ取得（httpd / secure / maillog / named）
2. parse_*.py で JSONL 化、時間窓フィルタ
3. analyzer.run() で Signal 生成
4. Mock + Claude 二段ふるい
5. pattern_tag から spoke skill を提示
6. 状況サマリ生成

§0 〜 §0.7 に **本番環境定数**（IP / パスワード / 触禁機器 / 既侵害前提 / 触らない哲学）が一元管理されており、全 spoke がここを参照する。

#### ⑥ 5 つの spoke skills

| skill | 対象 | 検出シグナル |
|---|---|---|
| `/wp-tamper` | victor & bravo | sqli/xss/cmdi/webapp 系 |
| `/dns-tamper` | bravo | dns/* |
| `/ddos` | 両機 | http-flood / dns-amplification / 同一 IP 多発 |
| `/phishing` | 両機 | mail/* |
| `/ransomware` | 両機 | privesc/* / persist/* / 不審 mtime |

それぞれ §1 で 18_統合脆弱性プレイブック由来の **検証済み観測コマンド** を実行、§3 で Claude 投入用プロンプト、§5 で報告書テンプレ生成、§6 で緊急対応コマンド雛形（**全部リーダー承認後**明記）。

#### ⑦ settings.json — 物理ゲート

3 層防御:

| 層 | ルール数 | 例 |
|---|---|---|

| **allow** | 64 | `ssh manage@10.1.1.{1,2}` `dig` `tail` `ss` `ps` `last` `journalctl` |

Claude が判断ミスしても、**実機への実行段階で物理的に止まる**。

### 3.5 設計判断 5 つの「なぜ」（5 分）

#### Q1: なぜ 60 項目を全部 pattern 化しなかったか

過去環境（com6.local / 2024 流用）固有の CVE に過学習すると、本番（com1.local）で別の CVE が出たとき検知ゼロになる。**6 カテゴリに圧縮した汎用 pattern**で「rainloop 1.12.0 でも roundcube でも捕捉する」形にした。18_ 自身も §59 で「攻略手順ではなく観点として活用」と明記している。

#### Q2: なぜ Mock + Claude の二段か

- Mock 単独 → 未知攻撃を取りこぼす
- Claude 単独 → API コストとレイテンシで 30 分制約を破る

**速い (Mock) と賢い (Claude) を両方使う**のが現実解。Mock filter で 8 割減、grey だけ Claude。これで予選 1 日のコストが現実的範囲に収まる。

#### Q3: なぜ Claude Code を選んだか（自前実装ではなく）

Claude Code は **slash command** で .md ファイルを実行手順として読める。Bash ツールで ssh / curl / dig 全部叩ける。Edit ツールでファイル操作も。**インシデント対応に必要な道具が全部揃っている統合環境**。自前で同じものを作ると 1 ヶ月かかる。

#### Q4: なぜ ControlMaster で SSH か

PDF「上記項目は変更しない」に従うと、サーバ側 authorized_keys 改変は NG。Mac 側だけで完結する **ControlMaster + ControlPersist 8h** が安全と速度のバランス。1 日 4 回手入力するだけ。詳細は [20_VPN_SSH接続手順_5_5.md](20_VPN_SSH接続手順_5_5.md)。

#### Q5: なぜ deny / ask / allow の 3 層か

「絶対実行しない」「人間判断要」「即実行」の 3 段階で粒度を分けることで、**安全と速度を両立**。read-only な dig / tail を毎回確認していては時間が足りない。一方で `iptables -j DROP` を allow にしては事故る。

### 3.7 限界（誠実に伝える）

このシステムが **できないこと**:

- **未知の攻撃カテゴリ** — 6 カテゴリの外（例: 0day の OS-level 脆弱性）は人間判断
- **最終的な対策実行** — Claude は提示まで、実行は人間が責任を持つ
- **電話応対の温度感調整** — 顧客の声色や苛立ち度の判断は人間
- **チーム間調整** — 他チームに迷惑をかけないかの判断
- **運営とのコミュニケーション** — リーダー判断の領域
- **本番環境の実機テスト未済** — 5/5 当日が初運用

### 3.8 思想（30 秒）

> **「触らない哲学 × 速い判定 × 物理ゲートで安全」** の三位一体。
>
> AI に任せるのは「速くて飽きない部分」だけ。**判断・責任・コミュニケーションは人間**。AI が暴走しないよう **物理ゲートで止める**。これが共有 demo 環境で AI を使う唯一の安全な形。

## §5. 安全ゲート発動シナリオ 2 例

### 例 1: BIND 設定編集を Claude が提案

```
Claude: 「Edit /etc/named.conf で allow-update を絞ります」
              │
              ▼
   settings.json deny ルール照合:
     "Edit(/etc/**)"  ← match!
              ▼
   ❌ 拒否 (実行不可)
              │
              ▼
   Claude: 「直接編集が拒否されました。代替手順:
              1. リーダー承認を取得
              2. cp -p で named.conf をバックアップ
              3. 別途 root SSH で手動編集
              4. 出題シナリオを破壊しないか運営確認」
```

### 例 2: HTTP flood 対策で iptables -j DROP

```
Claude: 「Bash: iptables -I INPUT -p tcp --dport 80 -j DROP」
              │
              ▼
   settings.json ask ルール照合:
     "Bash(*iptables*-j DROP*)"  ← match!
              ▼
   ⚠️  人間に確認プロンプト
              │
              ▼
   人間:  リーダー承認 / 顧客通知 / スナップショット 確認後 → Yes
              ▼
   ✅ 実行 (人間が責任を持つ前提で進む)
```
