## A-2. 各シナリオの植え込み (順序推奨)

優先順 (依存関係考慮):

1. **base 共通脆弱性** (両機 / 全シナリオ前提):
   - victor: /home/obuchi 作成 + chmod 777
   - bravo: useradd toor + UID 0 二重化 (/etc/passwd 編集)
   - 両機: SELinux disabled / pf disabled / firewalld inactive (既存設定の維持)

2. **phishing シナリオ**:
   - victor: dnf install dovecot → disable_plaintext_auth=no, ssl=no, passdb=pam, 143/110 listen ON
   - victor: sendmail インストール (デフォルトで 25/tcp listen)
   - bravo: courier-imap 同等の平文 IMAP (Rocky なので Dovecot で代用、後で FreeBSD への切替時に注意)
   - DNS に SPF/DKIM/DMARC TXT を **設置しない** (デフォルト = 脆弱)

3. **dns-tamper シナリオ**:
   - bravo: dnf install bind bind-utils
   - /etc/named.conf に allow-update { 10.0.0.0/8; }; (TSIG なし) + dnssec-validation no
   - ゾーン com1.local を作成 (zone "com1.local" + dummy A レコード)
   - 検証: dig @bravo www.com1.local + nsupdate で書換テスト

4. **ddos の DNS amplification 部分**:
   - bravo の BIND に allow-query { any; }; + recursion yes (ANY クエリで返答増幅)
   - 検証: dig @bravo ANY com1.local | wc -l (応答サイズ大)

5. **ransomware シナリオ**:
   - victor: rpm -i polkit-... (CVE-2021-4034 PwnKit に該当する古いバージョン or pkexec SUID 維持)
   - victor: useradd obuchi + /etc/sudoers に "obuchi ALL=(ALL) NOPASSWD:ALL" を追加 (既侵害想定)

## B-4. 異常検知ルール (出力末尾で flag)

- 既知サービスが inactive → 🚨 落ちてる (受電内容と整合確認)
- load average > CPU コア数 × 3 → 🚨 高負荷 (DDoS 疑い)
- /var/log 使用率 > 90% → ⚠️ ログあふれ
- 直近 1h 以内の /etc 配下変更 → 🚨 改ざん疑い (具体ファイル名を出す)
- listen ポートに想定外 (8080 / 31337 等) → ⚠️ 不審サービス

## 補足メモ (Claude には見せなくていい)

### なぜ 4 タスクをまとめて 1 セッションでやるか

- A (脆弱化) は OCI への ssh 作業、B (preflight) は .md 1 本作成、C (HTML 更新) は Edit 系、D は smoke test。**全部 read-write 系で別チャットに分けると context が分断**される
- A の植え込みが終われば D で即動作確認できる流れ
- C (HTML 更新) も B (/preflight 仕様確定) の後でないと書けないので、同セッションが効率的

### 想定所要時間

| タスク | 所要 |
|---|---|
| A (脆弱化) | 1-2 時間 (各サービス install + 設定 + 検証) |
| B (/preflight 作成) | 1 時間 (.md 1 本 + settings.json 追記) |
| C (HTML 更新) | 30 分 |
| D (smoke test) | 30 分 |
| 合計 | 3-4 時間 |

### 失敗時のフォールバック

- A で BIND が立ち上がらない → bravo を Rocky のままで dns-tamper シナリオは「設定ファイルだけ作る、サービスは止めたまま」で /check:check-bind-allow-update が静的検査として動く形に妥協
- B で /preflight が長すぎる → victor 部分だけ実装、bravo (FreeBSD) は本番直前に追加
- C は最低限 §3 への /preflight 追加だけで OK、§4 例の差し込みは省略可
- D は最低 D-2 (/preflight 空打ち) まで動けば本番投入 OK
