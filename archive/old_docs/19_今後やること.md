## 0. なぜこのドキュメントを作ったか（背景）

これまでの議論で次が判明した:

1. `.claude/commands/` の 6 本（`/incident` + 5 spoke）は型は揃っているが、**「実環境では動かない箇所」「権限ゲート未実装」「触禁機器ルール未反映」** がある
2. 危険コマンド（`iptables -j DROP` / `postsuper -d` / `usermod -L` / `chmod -s` / `kill -9` / `rm -rf` / named.conf 編集）が **「Claude が提示 → 人間が脊髄反射で実行」** で発火する事故動線がある
3. **デモ環境で予行演習せずに本番投入するのは事故の温床**。最低でも Docker レベルの再現環境で 1 周まわす必要がある
4. テスト環境ログ（`shirahama_test_log{,2,3}.md` 計 8,985 行）はあるが **「ベースライン（攻撃なし）」** で、analyzer の **誤検知ゼロ確認**には使えるが攻撃検知の検証には別途攻撃ログ生成が要る
5. 本番固有値（ドメイン `com1.local`、IP `10.1.1.x`、触禁機器 `10.1.130.1`、報告先 PukiWiki `133.42.49.140` 等）が commands に未埋め込み

これら全部を **48 時間後の予選までに何を潰し、決勝までに何を仕上げるか** を本ドキュメントで一元管理する。

### 3.4 テスト環境ログを analyzer に通す（誤検知ゼロ確認） 🔴

**目的**: 既存 3 本のテスト環境ログ（攻撃なしのベースライン）を analyzer に投入し、**誤検知が 0 件であること** を確認する。1 件でも検知が出るなら、それは false positive で、本番で同種のノイズが Claude を消耗させる。

**手順**:

1. `tests/fixtures/test_env_baseline/` を作成（後述 §7 でスクリプト雛形を用意済）
2. `shirahama_test_log{,2,3}.md` から `cat /var/log/<file>` 直後のブロックを抽出して .log ファイル化
3. `parse_*.py` でパース → JSONL
4. analyzer に通して signals 数を確認
5. 1 件でも出たら原因を調べる:
   - パターンが誤検知（例: WP の正規 URL を XSS と誤認）→ パターン修正
   - そもそもパースエラー → parser 修正
   - whitelist 漏れ → Mock backend に追加

**期待結果**: signals = 0 件、または「low severity の不可避ノイズのみ」。

### 4.3 Mock backend の whitelist 拡充 🟡

配布アカウント由来の活動を全 drop:

```python
# agent/backends/mock_backend.py に追加
KNOWN_GOOD_USERS = {"manage", "root", "admin", "vty", "enable"}
KNOWN_GOOD_IPS = {
    # チーム自身の VPN 接続元（決まっていれば）
    # 本番 booth11-15 の DHCP リース範囲 10.1.11.50-99
}

def is_known_good(signal):
    if signal.user in KNOWN_GOOD_USERS:
        return True
    if signal.src_ip in KNOWN_GOOD_IPS:
        return True
    return False
```

これで Claude API への流量が劇的に減る。

## 7. テスト環境ログを使った検証手順（今すぐ着手可能）

### 7.0 テスト環境ログの実態（重要 / 認識合わせ）

`shirahama_test_log{,2,3}.md` を抽出してわかったが、これは **`/var/log` の中身ではなく、`/etc/*.conf` などの設定ファイル偵察** がメイン。攻撃ログは含まれていない。

含まれているもの（30 ブロック / 1 ファイルあたり）:

- 設定ファイル: `resolv.conf` / `hosts` / `crontab` / `httpd.conf` / `named.conf` / `mysql my.cnf` / `dhcpd.conf` / `aliases` / `rainloop_index.php` / etc.
- バージョン: `victor__var_www_wordpress_wp-includes_version.php`、`rainloop/data/VERSION`
- OS: `victor__etc_os-release` / `redhat-release`
- DHCP リース: `var_lib_dhcpd_dhcpd.leases`

**したがって、用途は 2 つに整理される**:

| 用途 | やること | 価値 |
|---|---|---|
| **(A) 環境一致検証** | 14_レポートの前提（PHP 7.2.24, WP 4.9.4, BIND `allow-update {10.0.0.0/8}` 等）が現環境と乖離していないか cross-check | 高（commands の前提が正しいか） |
| **(B) commands の参照パス検証** | `.claude/commands/` で参照しているパス（例: `/usr/local/etc/namedb/named.conf`）が実在するか確認 | 高（本番でファイル not found を防ぐ） |

**(C) analyzer 誤検知ゼロ確認** は **このログでは できない**。配信されるログ（access_log / secure / maillog / named.log）は別途取得が必要。当面は **§5.1 Docker デモ環境** 構築後に取得する。

### 7.1 ログの抽出

抽出スクリプト `tests/extract_test_logs.py` を用意済。

```bash
cd /Users/ryu/Desktop/shirahama

# 3 本連続で実行（2 本目以降は --append）
python tests/extract_test_logs.py docs/shirahama_test_log.md  tests/fixtures/test_env_baseline/
python tests/extract_test_logs.py docs/shirahama_test_log2.md tests/fixtures/test_env_baseline/ --append
python tests/extract_test_logs.py docs/shirahama_test_log3.md tests/fixtures/test_env_baseline/ --append

# 確認
ls tests/fixtures/test_env_baseline/ | head
```

→ `bravo__etc_resolv.conf`, `victor__etc_httpd_conf_httpd.conf` 等が 30 ファイル前後できる。

### 7.2 環境一致検証（(A) の使い方）

抽出した設定ファイルから、`14_サーバ調査レポート` の前提が変わっていないか確認:

```bash
# WordPress バージョンが 4.9.4 のままか
cat tests/fixtures/test_env_baseline/victor__var_www_wordpress_wp-includes_version.php | grep wp_version

# rainloop が 1.12.0 のままか
cat tests/fixtures/test_env_baseline/victor__var_www_rain_rainloop_rainloop_v_wildcard_VERSION

# BIND の allow-update が「10.0.0.0/8」 のままか（致命的設定の確認）
cat tests/fixtures/test_env_baseline/bravo__usr_local_etc_namedb_named.conf | grep -A2 'allow-update'

# DHCP の not authoritative がまだ残っているか
cat tests/fixtures/test_env_baseline/victor__etc_dhcp_dhcpd.conf | grep -i authoritative
```

→ 想定と乖離している場合は、commands の前提と `14_レポート` を更新する。

### 7.3 commands 参照パスの実在検証（(B) の使い方）

`.claude/commands/*.md` 内で参照されているパス（`/usr/local/etc/namedb/named.conf` 等）が、抽出ファイル名と一致するか目視確認:

```bash
# .claude/commands/ で言及されている /usr/local/, /etc/, /var/ 配下のパスを抽出
grep -hoE '/(usr/local|etc|var)/[^ '\''"`)]+' .claude/commands/*.md | sort -u

# 抽出されたファイルの方は
ls tests/fixtures/test_env_baseline/ | sed 's/^[^_]*__//; s/_/\//g; s/^/\//'
```

→ commands で言及があるが抽出に存在しないパスは、**そもそもサーバ上に無い** か **アクセス権が必要だった** 可能性がある。要再調査リストに入れる。

### 7.4 analyzer 誤検知ゼロ確認（攻撃ログは別途必要）

実ログ（access_log / secure / maillog / named.log）は **本テスト環境ログには無い** ので、これは:

- 予選 Day1 で取った最初の `/incident` ログを保存
- §5.1 Docker デモ環境を立てた後にログを生成

の 2 つのルートで取得する。
