電話対応 / 15–60分

2本目以降の電話に備え、既存の通話者と温度を保つ。参照: 01 / 04
  • ✓
    15〜30分ごとに顧客へ途中報告の連絡
    「調査継続中」「影響範囲はまだ確定していない」と事実で伝える。
  • ✓
    追加受電があれば既存事案と紐づける
    同じ現象なら同じチケット番号にまとめる。別現象なら別番号。
  • ✓
    2件目の陽動を想定してヘルプを要請
    電話が被ったら記録係が応援。指揮に優先度を判定させる。
  • ✓
    経営層からの問い合わせは指揮経由で
    技術の一次情報を指揮が整形してから回す。
  • ✓
    復旧時刻を約束しない
    「次の連絡時刻」だけ約束する。例: 「30分後に状況をお電話します」
  • ✓
    外部通報の話は本大会では扱わないと明確に
    警察・IPA・個情委の話題が出たら「社内プロセスで確認」と返す。

指揮 / 15–60分

封じ込めの判断を下す。記録と連絡線を維持する。参照: 02.1 / 03 / 05
  • ✓
    仮説を1つに絞らず、上位3つを並べる
    例: DDoS / DB詰まり / DNS。優先的に潰す順を決める。
  • ✓
    03 シナリオ別プレイブックから該当項目を開く
    手順を逸脱しない。迷ったら一番影響が限定的な手から。
  • ✓
    封じ込めの GO/NO-GO を言語化
    「誰が・何を・いつ・どこまで」遮断するかを口頭で全員に通知。
  • ✓
    影響範囲を図で更新
    どのサーバが正常 / 影響 / 停止中かを色分け。
  • ✓
    ログ保全指示を出す
    再起動やローテート前に対象を退避。「何を」「どこへ」を指定。
  • ✓
    05 全体チェックの Phase 2 欄を埋めていく
    終わった項目を可視化。漏れを防ぐ。
  • ✓
    30分ごとに1分ミーティング
    進捗・ブロッカー・次の15分の目標だけ。
⚠ 「封じ込め > 復旧」の順序を崩さない。先に戻すと再発して損害が広がる。

サーバー / 15–60分

ログを深掘りし、原因仮説に白黒をつける。参照: 02.1 / 03 / 06
  • ✓
    アクセスログのエラー比率・レイテンシ分布
    例: 5xxの増加・TTFBの肥大。数値をメモ。
  • ✓
    関連サービスとの依存関係を洗う
    DB / キャッシュ / 外部 API / 認証基盤。障害点は隣のサーバの可能性。
  • ✓
    プロセス単位の挙動を記録
    pidstat -u 1 / vmstat 1 でスパイクか定常かを判別。
  • ✓
    改ざん / 侵害の痕跡をチェック
    新規ファイル・cron の不審エントリ・SSH 公開鍵・uid=0 の新規ユーザ。
  • ✓
    封じ込めの実行 (指揮の合図後)
    例: 問題プロセスの隔離、疑いアカウント無効化、不要ポートのクローズ。
  • ✓
    証拠保全: ログ・メモリ・ディスクのコピー
    上書きされる前に退避。ハッシュも取っておく。
  • ✓
    設定変更は diff を残す
    復旧後に戻せるよう、前後の diff と実行時刻を記録。
⚠ OS差異: RHEL 系は /var/log/httpd/, Debian 系は /var/log/apache2/, FreeBSD は /var/log/httpd-*.log。

ネットワーク / 15–60分

遮断と迂回の実施。影響の二次監視。参照: 06
  • ✓
    攻撃源の特徴量を決めてから遮断
    IP / AS / UA / パス / レート。根拠を1つに絞って書く。
  • ✓
    ACL / WAF ルールを上書きでなく「追加」
    YAMAHA ip pp secure filter は上書きされやすい。差分のみ追加する運用。
  • ✓
    AWS: WAF Rate-Based / SG / NACL の優先順位を整理
    Rate-Based を先、恒久対策として SG、広域遮断は NACL。
  • ✓
    遮断後の「正常側」への影響をモニタ
    誤遮断が起きていないか。ヘルスチェックや内部通信。
  • ✓
    DNS 切替の是非を検討
    TTL・切替先の可否・戻し条件を先に決める。
  • ✓
    スループット・セッション数の時系列を保存
    後のレポートと振り返りに必須。スクリーンショットで可。
⚠ 遮断の効果は2–5分で現れる想定。即効性がなければ「観測点のズレ」を疑う。
