# /ticket — PukiWiki トラブルチケット自動生成

引数: `[<triage-report-path>]`
省略時: `~/Desktop/shirahama_test.md`

例:
- `/ticket` (デフォルトパス)
- `/ticket ~/Desktop/incident_2026-05-05_12-30.md`

## このスキルが解決する問題

大会の最後 30 分は **PukiWiki に 1 件 1 チケット** で事案記録を投稿するフェーズ。`/report` (顧客向け完了連絡) は社外向け文面なので、**運営/評価者向けの技術記録** には使えない (記載粒度・エビデンス引用・読み手が違う)。

本スキルは **トリアージ報告 + 対応記録 → PukiWiki マークアップ 1 件** を自動生成する。

評価軸「**チケット記載の正確さ・網羅性**」(sccs2026_二次予選まとめ.md L.184) に直接対応。

---

## §0. 前提

- 入力: `/incident` `/check:*` `/playbook:*` の出力をまとめた Markdown (shirahama_test.md 同等構造)
- 出力: **PukiWiki マークアップで画面表示**。投稿先 `http://133.42.49.140/trouble_ticket_137/` (whiskey / E5mA9cF3) — 投稿は人間が手で行う (認証情報を Claude に渡さない方針)
- 1 回の呼び出しで **1 インシデント分** のチケットを生成。複数件ある場合は引数を変えて複数回呼ぶ

---

## §1. 入力読み込み

```bash
SHIRAHAMA_DIR="${SHIRAHAMA_DIR:-/Users/ryu/Desktop/shirahama}"
REPORT_PATH="${1:-$HOME/Desktop/shirahama_test.md}"

if [ ! -f "$REPORT_PATH" ]; then
    echo "❌ トリアージ/対応記録が見つからない: $REPORT_PATH"
    echo "   /incident → /playbook:* を先に実行するか、引数でパスを指定してください"
    exit 1
fi

echo "─── /ticket §1 入力 ───"
echo "  source : $REPORT_PATH ($(wc -l < $REPORT_PATH) 行)"
echo "  target : http://133.42.49.140/trouble_ticket_137/"
```

その後 Claude は **Read ツールで `$REPORT_PATH` を全文読む**。

---

## §2. 必須フィールド抽出 (Claude が実施)

PukiWiki チケットの 5 フィールド (sccs2026_二次予選まとめ.md L.171-176 推奨):

| フィールド | shirahama_test.md での出所 | 抽出ルール |
|---|---|---|
| **発生時刻** | §1 時間窓 / §2.2 攻撃時系列 | JST 24h 形式 (例: `2026-05-05 13:37`)。攻撃開始時刻を採用 |
| **トラブルの内容** | §2.2 + §6 統合解釈 | 1 段落 (3-5 文)。技術的事実中心、感情語禁止 |
| **原因** | §3 検出シグナル + §4 判定 + §5 副次発見 | **必ずログ行を引用** (どこから判明したか)。脆弱性根本原因を含める |
| **対処内容** | §8 推奨封じ手 (実施済のもののみ) + 対応セッション中の Bash 実行履歴 | 時系列、実施者・実施時刻・実施コマンド (commands を引用) |
| **復旧確認** | §7 突破ステータス + 検証結果 | どのログ/URL/コマンドで「正常状態」を確認したかを明記 |

抽出時の注意:
- **「リーダー承認後」止まりで未実施の封じ手はチケットに書かない** (= 嘘になる)
- **エビデンスがないものは「(調査継続中)」と明記** — 推測で埋めない
- **時刻は必ず JST** (UTC で記録されているなら +9h 変換)
- **副次発見 (toor / 777 home 等) は別チケットの候補として末尾に「関連事項」で添える**、本体には混ぜない

## §3. PukiWiki マークアップ生成

### 3.1 出力テンプレ (Claude はこれを実データで埋める)

```pukiwiki
*トラブルチケット: {1 行サマリ}
#contents

**発生時刻
{YYYY-MM-DD HH:MM JST} (検知: {検知時刻})

**トラブルの内容
{1 段落、技術事実中心、3-5 文}

**原因
{根本原因 1-2 文}。

''関連ログ (一次エビデンス):''
 {ログ行の引用 (行頭スペース 1 個でコードブロック化)}
 {追加引用があれば複数行}

**対処内容
|時刻 (JST)|実施者|実施内容|コマンド/操作|h
|HH:MM|{役割}|{何をしたか}|{コマンド or 操作概要}|
|HH:MM|{役割}|{...}|{...}|

**復旧確認
- 確認方法1: {コマンド or URL} → 期待値: {何が正常か} → 結果: {正常/未確認}
- 確認方法2: {...}

**影響範囲
|項目|値|h
|サービス停止|{なし or 期間と対象}|
|情報漏えい|{現時点で確認できる範囲では未確認 or 確認済の内容}|
|影響利用者|{人数 or 範囲}|

**関連事項 (別チケット候補)
- {副次発見 1: 例 - bravo に toor UID0 backdoor、別途検査要}
- {副次発見 2}

----
記載者: {対応者名} / 記載時刻: {生成時刻 JST}
```

### 3.2 PukiWiki マークアップ規約 (このスキルが守る)

| 要素 | 構文 | 用途 |
|---|---|---|
| 見出し | `*` `**` `***` | h1/h2/h3 |
| リスト | `-` `--` | 通常 / ネスト |
| テーブル | `|...|...|` 末尾 `h` でヘッダ行 | 時系列・影響範囲 |
| 強調 | `'''bold'''` `''italic''` | 引用ラベル等 |
| コードブロック | 行頭半角スペース 1 個 | ログ引用 |
| 罫線 | `----` | セクション区切り |
| 目次 | `#contents` | 自動目次 |

**禁則**: HTML タグ (`<br>` 等) は使わない / 絵文字は使わない (公式記録なので `🚨` `✅` 等は避ける、代わりに `[要対応]` `[確認済]` 等の角括弧テキスト)。

### 3.3 1 行サマリの書き方

```
{攻撃カテゴリ} from {攻撃元 IP} {突破有無} ({標的})
```

例:
- `WP/IMAP brute force from 161.33.12.212 未遂 (victor wp-login + bravo dovecot)`
- `DNS amplification reflection 踏み台化阻止 (bravo BIND)`
- `WP 改ざん復旧 (victor /wp-content)`

## §4. 完了判定 (チケット投稿可否)

以下を Claude が確認し、画面冒頭に表示:

```
─── /ticket §4 投稿可否判定 ───
[✅ or ❌] 発生時刻が JST 24h で書ける
[✅ or ❌] 原因にログ引用が 1 行以上ある
[✅ or ❌] 対処内容が時系列テーブルで書ける (実施済の項目が 1 件以上)
[✅ or ❌] 復旧確認の方法が 1 つ以上ある
[✅ or ❌] 影響範囲 3 項目すべてに値がある (「未確認」も値として OK)

判定: ✅ 投稿可 / ⚠️ 一部「調査継続中」で投稿可 / ❌ 投稿不可 (情報不足)
```

❌ の場合、不足項目をリストして **どの /check:* / /playbook:* をもう一度叩くべきか提示** (例: 「対処内容なし → /playbook:wp-tamper §6 を未実行、リーダー承認取って実行してから再生成」)。

## §6. 複数インシデント対応

1 セッションで複数インシデントを捌いた場合、**入力ファイルを分けて複数回 `/ticket` を呼ぶ**運用を推奨:

```
/ticket ~/Desktop/incident_1_wp-tamper.md
/ticket ~/Desktop/incident_2_dns-amplification.md
/ticket ~/Desktop/incident_3_phishing.md
```

`/incident` 起動時に出力ファイルをインシデント番号付きで分けておくと、本フェーズで楽。

## §7. NG パターン (このスキル自身の自戒)

- **未実施の封じ手を「実施した」と書かない** — `/report` と同じ自戒
- **ログ引用なしの「原因 = ○○」は禁止** — 評価軸「正確さ」に直撃
- **顧客名 / 社員名を書かない** — チケットは管理会社視点の技術記録、個人名は不要
- **「絶対に」「完全に」断定しない** — 「現時点で確認できる範囲では」を必須
- **絵文字 (🚨 ✅) を使わない** — PukiWiki が崩れる + 公式記録としては不適
- **副次発見を本体チケットに混ぜない** — 別チケット候補として「関連事項」セクションに分離
- **§0.6 「触らない哲学」と整合**: チケット投稿は人間、Claude が curl で投稿しない (認証情報を渡さない)

## 8. JSON 永続化（HTML dashboard 連携）

§3 で生成した PukiWiki マークアップ + §2 抽出フィールドを JSON 化して helper に渡す。actor は `ai_human` (生成は AI、投稿は人間)。

```bash
cat <<'JSON_EOF' | scripts/emit_skill_json.sh ticket --actor ai_human
{
  "inputs": {
    "triage_report_path": "<§1 で読み込んだ shirahama_test.md 等のパス>"
  },
  "outputs": {
    "fields": {
      "occurrence_time": "<発生時刻 (JST)>",
      "trouble_content": "<事象内容>",
      "cause": "<原因 (確定 / 推定の区別を明記)>",
      "actions_taken": ["<対応内容を時系列で>"],
      "recovery_verification": "<復旧確認手順 + 結果>"
    },
    "pukiwiki_markup": "<§3.1 で生成したマークアップ全文>",
    "submission_url": "http://133.42.49.140/trouble_ticket_137/index.php"
  },
  "verdict": {
    "status": "info",
    "summary": "PukiWiki マークアップ生成完了。投稿は人間が手で実施"
  },
  "next_skills": []
}
JSON_EOF
```

- `actor=ai_human` で「生成は AI、投稿は人間」を明示 (§0.6 触らない哲学と整合)
- `pukiwiki_markup` には §3.1 のテンプレ全文をそのまま入れる (改行は `\n` でエスケープ)
- 保存先: `data/incidents/${INCIDENT_ID}/ticket__<ts>.json`
