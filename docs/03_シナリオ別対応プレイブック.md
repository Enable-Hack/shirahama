# 03. シナリオ別対応プレイブック

> 技術調査の結果、インシデント内容が特定できた段階で参照する **シナリオ別の対応手順集**。
> 大会で出題が想定される代表シナリオをカバー。各シナリオは **判定 → 封じ込め → 復旧 → 顧客報告** の4ステップで統一。

## シナリオ1：フィッシング（認証情報詐取）

### 症状・入電パターン
- 「メールのリンクをクリックしてID/パスワードを入力してしまった」
- 「社内ポータルそっくりのページで入力した後、画面が真っ白になった」
- 外部からの通報：「貴社社員名義で怪しいメールが届いた」

### 判定ポイント

| 確認事項 | 方法 |
|---|---|
| 入力したサイトのURL | ユーザから正確に聞き取り（可能ならスクショ） |
| 入力した情報 | ID, パスワード, 多要素コード, 個人情報のどれか |
| 入力時刻 | ログイン監視・認証ログと突き合わせ |
| メールの送信元・件名 | メールヘッダを確保 |
| 同一メール受信者 | メールサーバで件名・送信元で検索 |

### 封じ込め（技術）

1. **対象アカウントのパスワード強制リセット**（AD／認証基盤）
2. **該当アカウントで進行中のセッションを全切断**（VPN, Web, メール）
3. **該当メールを他ユーザの受信箱からも削除**（管理者権限での一括削除）
4. **送信元ドメイン／URL を FW・メールサーバでブロック**
5. **多要素認証の再登録**（OTPトークンやアプリを再発行）

### 復旧・影響確認

- 対象アカウントの **認証ログ監査**（過去7日、外部IPからのアクセス有無）
- メールサーバの **送信ログ監査**（当該アカウントからの不審送信）
- 社内共有ドライブ・クラウドでの **アクセス履歴**
- 派生アクセス（ADトラストによる他システム侵入）の確認

### 顧客・ユーザへの報告テンプレ

```
【一次報告】
〇〇様のアカウントについて、フィッシングサイトへの認証情報入力を確認しました。
・10:20 該当アカウントをロックし、パスワードをリセット済み
・10:25 当該アカウントでの外部不審アクセスは現時点で観測されていません
・10:30 同メールを受信した他の社員から、さらなる被害報告はありません
引き続き認証ログを監視し、11:00 に次回報告いたします。
```

### NG対応
- ユーザに「大丈夫です」と断定する（外部アクセスの有無確認前）
- 「パスワードを教えて」とこちらから聞く
- 該当メールを「そのまま放置してOK」と伝える（他ユーザが開封する恐れ）

## シナリオ2：WordPress 改ざん・不正アクセス

### 症状・入電パターン
- 「サイトのトップページが別画像になっている」
- 「WordPress の管理画面に入れない」「身に覚えのない記事が投稿されている」
- 「アクセス解析で海外IPから大量アクセス」

### 判定ポイント

| 確認事項 | 方法 |
|---|---|
| 改ざん箇所 | ブラウザ＋サーバ側ファイル両方で確認 |
| WP バージョン・プラグイン | `wp-admin` の一覧 or `wp-cli` |
| 直近の変更 | `wp-content/` 配下の mtime 確認 |
| 管理ユーザ | `wp_users` テーブルに不審ユーザがいないか |
| REST API 悪用痕跡 | access_log で `/wp-json/wp/v2/users` `/wp-json/wp/v2/posts` 等へのアクセス |
| 認証ログ | wp-login.php への総当たり痕跡 |

### 封じ込め（技術）

1. **WPサイトを一時的にメンテナンスモード／IP制限**（.htaccess 等で応急ブロック）
2. 不審な **管理ユーザを削除**、既存管理者の **パスワード全員リセット**
3. `/wp-content/uploads/` 等に **PHPバックドアが配置** されていないか確認・削除
   - 例：`find /var/www/ -name '*.php' -newer /tmp/ref -type f`
4. **REST API を制限**（認証必須化、`users` エンドポイント無効化）
5. プラグイン・テーマを **最新版へ更新**、未使用は削除

### 復旧

- 正常なバックアップから **ファイルのみ** を復元（DB は汚染確認後に判断）
- 汚染箇所を特定できていれば **該当ディレクトリ単位で差し替え**
- 改ざん検知を入れる（AIDE／簡易 md5 リスト）

### 顧客への報告テンプレ

```
【一次報告】
公開Webサイトの改ざんを確認しました。
・原因調査中：WordPress REST API 経由の不正投稿が有力
・10:40 サイトをメンテナンスモードに切替（閲覧者には告知ページを表示）
・10:45 不審な管理ユーザを削除、全管理者パスワードをリセット
・次のアクション：バックアップからの復元を実施予定（実施前に再度ご連絡）
想定復旧時刻：11:30
```

### NG対応
- 「サーバを再起動すれば直ります」
- 原因不明のままバックアップ復元（同じ経路で再感染する）
- 「管理者パスワードを変えてください」とユーザに丸投げ（こちらで強制実施すべき）

## シナリオ3：ランサムウェア感染疑い

### 症状・入電パターン
- 「ファイルが開けない」「拡張子が変わった」「見たことない文字の脅迫文が出た」
- 「ネットワークドライブのファイルまで開けない」
- 「デスクトップに README-*.txt がある」

### 判定ポイント（**冷静に、断定を急がない**）

| 確認事項 | 方法 |
|---|---|
| 暗号化されたファイル | 拡張子パターン／中身のマジックナンバー |
| 脅迫文 | 本文を写真で確保（クリックはさせない） |
| 感染端末の範囲 | ファイルサーバ上の暗号化ファイル mtime から逆算 |
| 感染起点 | 最も早い mtime の端末／ユーザ |
| 横展開経路 | SMB, RDP, 管理共有, AD資格情報 |

### 封じ込め（**封じ込めが復旧より優先**）

1. **該当端末を LAN から物理切断**（電源は **切らない**：揮発メモリ保全）
2. **ファイルサーバへの書き込みを即停止**（共有を read-only に or 切断）
3. **AD の侵害想定**：管理者アカウントのパスワード一斉変更、krbtgt の二重リセット検討
4. **VPN／リモート経路の遮断**
5. **バックアップ媒体を物理的に隔離**（NASへのパスをアンマウント、LTOはオフライン保管）

### 被害範囲特定

- ファイルサーバの暗号化ファイル mtime 最古 → 感染起点時刻
- AD の認証ログ・成功した RDP／SMB ログイン
- ウィルス対策ソフト・EDR の検知履歴
- メールログ：感染起点日前後の添付ファイル付きメール

### 復旧

- 感染端末は **初期化前提**（マルウェア残置の可能性、クリーン化は困難）
- ファイルサーバは **バックアップから復元**、復元前に **経路の封じ込め完了** を確認
- **復元直後に再感染しないよう**、AD 資格情報と脆弱性（SMBv1、未パッチ OS 等）を事前に潰す

### 顧客への報告テンプレ

```
【一次報告】
ファイルサーバの一部ファイルについて、ランサムウェア疑いの症状を確認しました。
・10:30 感染疑いの端末（営業部PC-3台）を LAN から切り離しました
・10:35 ファイルサーバを読み取り専用に変更し、書き込みを停止しました
・10:40 AD管理者のパスワードを一斉変更しました
・被害範囲：営業部共有フォルダの約〇〇ファイル（暗号化確認済み）
・次のアクション：バックアップからの復元可否を確認中
・復旧見通し：現時点では未確定、11:00 に改めて状況をご報告します
```

### NG対応
- 脅迫文のリンクを **クリックして内容確認**
- 「身代金を払えば戻ります」と助言（大会的にも現実的にも NG）
- 感染端末の **電源即OFF**（揮発メモリ・プロセス情報が消える）
- 復元を急ぎすぎて **経路未封じのまま書き戻す**（即再暗号化される）

## シナリオ4：DDoS／大量アクセス

### 症状・入電パターン
- 「サイトが重い／タイムアウトする」
- 「監視で応答時間が跳ね上がっている」
- 「特定IPから秒間数百リクエスト」

### 判定ポイント

| 確認事項 | 方法 |
|---|---|
| アクセス元の分布 | access_log, FW/UTMログ |
| 単一IP / 分散 | 集中型か DDoS型か |
| ターゲット | 特定URL／全体／認証エンドポイント |
| レイヤ | L3/L4（SYN flood等） or L7（HTTP flood） |
| 実ユーザ影響 | 正常トラフィックの応答時間 |

### 封じ込め

- **単一／少数IP** → FW・UTMでIP遮断
- **分散・大量** → 上流のISP／CDN／WAF に絞り込み依頼（大会想定では UTM 側ACLで対処）
- **L7攻撃** → レート制限、User-Agent／リファラで絞り込み
- **正規ユーザ保護** → 重要経路は別VLAN・別サーバにフェイルオーバ

### 判断

- 攻撃トラフィックが **DNS増幅・SYN flood** 系 → 上流側対応が必要、社内完結は困難
- **特定ディレクトリへの集中** → アプリレイヤの対策（認証前キャッシュ等）
- 陽動の可能性：**同時刻に別の侵入痕跡** がないか必ず確認

### 顧客への報告テンプレ

```
【一次報告】
公開サイトへの大量アクセスを観測しています（10:15 より）。
・攻撃元：海外数十IPから分散、秒間○○リクエスト
・影響：一般ユーザの応答遅延、10:25 時点でページ表示まで○○秒
・10:30 UTMで攻撃元上位10IP群を遮断、応答速度は改善傾向
・並行確認：同時刻に他システムへの侵入痕跡がないか監視強化中
・次回報告：11:00
```

### NG対応
- 監視アラートを止めて「収まった」と判断
- DDoS に気を取られて **同時進行の別侵入を見逃す**
- サーバスペックを上げるだけで対処（根本解決にならない）

## シナリオ5：マルウェア感染端末（ランサム以外）

### 症状・入電パターン
- 「ウィルス対策ソフトが警告を出した」
- 「PCの動作が急に遅い」「見知らぬプロセスが動いている」
- 「勝手にブラウザが開く・広告が出る」

### 判定ポイント

| 確認事項 | 方法 |
|---|---|
| 検知名・ハッシュ | AV/EDR の検知ログ |
| 感染経路 | メール添付／USB／Webダウンロード |
| 通信先 | プロセス→外部IPの関連付け |
| 他端末への影響 | 同一セグメント／同一ユーザの別端末 |

### 封じ込め

1. 端末を **LAN切離** （電源は切らない）
2. 同一 **認証情報** を使う他端末の状態確認
3. **C2通信先** を FW/UTM で遮断
4. メール経路なら **同メールの他受信者** を特定・隔離

### 復旧

- 検知・駆除で済むもの → AV/EDR の指示に従う
- **持続化・権限昇格を伴うもの** → 端末初期化を推奨
- 認証情報流出疑いがあれば **対象アカウントのパスワード変更**

### NG対応
- 「AV が駆除したから大丈夫」で終わる（横展開・残置の可能性）
- 本人にそのまま使わせ続ける

## シナリオ6：情報漏えい疑い（メール誤送信・設定ミス）

### 症状・入電パターン
- 「社外に機密情報をメールしてしまった」
- 「クラウドストレージの共有リンクを公開設定にしていた」
- 「個人情報の含まれたファイルがWebから見えた」

### 判定ポイント

| 確認事項 | 方法 |
|---|---|
| 漏えい対象 | 個人情報／機密情報／どちらでもない |
| 漏えい件数 | 行数・ファイル数・アカウント数 |
| 送信／公開時刻 | メールログ／アクセスログ |
| 外部で閲覧された痕跡 | Webログ、クラウド共有リンクのアクセス数 |

### 封じ込め

- **メール誤送信**：回収依頼／送信取消（Exchange 等の回収機能）、相手に削除依頼
- **クラウド共有リンク**：リンクを即無効化、公開範囲を再設定
- **Web公開**：当該ファイルを即削除、キャッシュ削除依頼（必要なら検索エンジン側）

### 顧客への報告

- **件数と内容を正確に**（推計と実数を区別）
- 外部で閲覧された痕跡の **ログ裏取り** を添えて報告
- 本大会スコープでは **外部機関への通知は扱わない**。顧客への正確な事実提供に集中

### NG対応
- 件数の曖昧な報告（「数十件くらい」）
- 漏えい有無の即断（まだ分からない段階で「漏れてません」）

## シナリオ7：Web 改ざん（静的ページ書き換え）

### 症状・入電パターン
- 「トップページに見たことない画像・メッセージが出ている」
- 「外部から『貴社サイトが改ざんされている』と通報」

### 判定ポイント

| 確認事項 | 方法 |
|---|---|
| 改ざん範囲 | 単一ファイル／全体／特定ディレクトリ |
| サーバ側ファイルの mtime | `find / -mtime -1 -type f` 等 |
| 侵入経路 | SSH認証ログ／FTPログ／アプリ脆弱性 |
| 改ざん内容 | 政治的主張／マルウェア埋め込み／SEO spam |

### 封じ込め

1. **即座に公開停止**（WebサーバのDocRootを差し替え or メンテナンスページ）
2. 侵入経路を特定するまで **復旧開始しない**
3. SSH／管理系アカウントのパスワード変更、不要アカウント削除
4. アップロード機能など **入力経路に脆弱性がないか** 確認

### 復旧
- **バックアップから復元**、復元前に脆弱性対応
- **改ざん検知** を導入（次回早期発見のため）

### NG対応
- 改ざん箇所だけ戻して「完了」（侵入経路を残したまま）
- 怪しいファイルを削除せず「上書きで直した」

## シナリオ8：不正送信メール（なりすまし／オープンリレー）

### 症状・入電パターン
- 「貴社ドメインから迷惑メールが来た」（外部からの通報）
- 「取引先から『送ってないメールが届いた』と連絡」
- 「メールサーバの送信キューが異常に詰まっている」

### 判定ポイント

| 確認事項 | 方法 |
|---|---|
| 送信元 | 本当に自社サーバ発か（メールヘッダの Received 連鎖）|
| なりすまし／実送信 | SPF/DKIM/DMARC 検査結果 |
| 自社サーバからの送信なら経路 | アカウント侵害／オープンリレー／内部マルウェア |
| 送信量 | メールキュー・ログから集計 |

### 封じ込め

- **アカウント侵害**：対象アカウント無効化・パスワードリセット、セッション切断
- **オープンリレー**：SMTPの relay 設定を即閉鎖、認証必須化
- **内部マルウェア**：該当端末を隔離
- 送信キュー内の **未送信分を停止**

### NG対応
- 「なりすましだから関係ない」で切る（**ヘッダ確認前に断定しない**）
- 送信量を確認せず「数件でしょう」と応答

## 共通：シナリオ対応の流れ（どのシナリオにも当てはまる）

```
① 判定    → 何のインシデントか、事実と推測を分けて確定
② 封じ込め → 拡大を止める（復旧より優先）
③ 調査    → 原因・経路・範囲の特定
④ 復旧    → 封じ込めと原因対処後にのみ実施
⑤ 報告    → 顧客／ユーザへ一次・中間・完了の3段階
⑥ 再発防止 → 経路を塞ぐ（パッチ・設定変更・監視追加）
```

### 優先順位の鉄則

1. **人命・安全** > **法令遵守** > **業務継続** > **データ保全** > **利便性**
2. **拡大防止** > **原因究明** > **復旧** （順番を間違えないこと）
3. **事実** > **推測** > **願望**（報告では必ずこの順）

## シナリオ不明時のデフォルト対応

> 判定がつかない場合でも **「何もしない」を避け** 、以下を同時進行で開始。

1. 対象端末／サービスを **観察モード**（触らず、ログだけ取る）
2. **ログ保全**（別媒体にコピー、上書き前に退避）
3. **時系列メモ** 開始（チーム内ホワイトボード）
4. **顧客への中間報告**：「現在調査中、◯分後に再度ご連絡」
5. 30分ごとにシナリオ再判定
