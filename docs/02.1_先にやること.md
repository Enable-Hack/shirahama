# 02.1. 先にやること — 電話を取ってから DDoS 判定に入るまで

> [01_受電ヒアリングシート](./01_受電ヒアリングシート.md) で話を聞いたあと、[06_DDoS対応詳細](./06_DDoS対応詳細.md) に入る前の **切り分けフロー**。
>
> ★ **電話で来るクレームの 8 割は DDoS ではない**。いきなり DDoS 対応に入ると、DB 詰まり・外部 API 遅延・DNS 障害などを誤診して遮断事故を起こす。**この章は飛ばさない。**

---

## 大原則：6ステップの順番を守る

```
電話 → 再現確認 → 環境把握 → 構成把握 → ログ確認 →（ここで初めて）DDoS 判定
```

どれかを飛ばしたら判定は信用できない。

---

## 最重要な視点：観測点 ≠ 攻撃対象 ≠ 被害箇所

これを最初に頭に叩き込む。

| 用語 | 意味 |
|---|---|
| **観測点** | 症状が最初に見えたサーバ（＝ユーザが「おかしい」と言ったもの） |
| **攻撃対象** | パケットが実際に届いているサーバ |
| **被害箇所** | リソースが先に枯渇する本当のボトルネック |

この3つはズレる。**被害箇所で遮断しても根本解決にならない**（例：Web に来た HTTP Flood で DB が詰まった時、DB 側で制限をかけると正規クエリも止まる）。

---

## Phase 0：電話（事実と事象の分離）

ユーザの言葉を「事実」と「事象（主観）」に分ける。

```
【事実】（数字・時刻・URL・IP）
・10:15 から /login にアクセスすると 30 秒待っても返らない
・同じ部署の 3 人が同じ症状
・営業部の PC だけ。経理部は問題なし

【事象（主観）】
・「サイトが重い」      ← どれくらい？どのページ？
・「メールが来ない」    ← いつ送った？誰から？
・「全部ダメ」          ← 本当に全部？何を試した？
```

聞き取りの原則：**「いつから・何が・どう・どこで」を数字と固有名詞で埋める**。

## Phase 1：再現確認（30秒）

自分の手で同じことをやって、どのレイヤで失敗するか見る。

```bash
# ① HTTP レイヤ
curl -I -m 5 https://対象URL

# ② TCP レイヤ
nc -zv 対象ホスト 80
nc -zv 対象ホスト 443

# ③ DNS レイヤ
dig +short 対象ホスト
nslookup 対象ホスト

# ④ L3 到達
ping -c 3 対象ホスト
traceroute 対象ホスト    # 経路途中での切断を見る
```

### 判定表

| 結果 | 判定 |
|---|---|
| curl 200 OK | ユーザ環境側の問題（キャッシュ / DNS / プロキシ） |
| curl タイムアウト | サーバ or 経路の問題 → 次フェーズへ |
| curl 5xx | サーバ稼働 + アプリ障害（DDoS 以外の可能性も高い） |
| curl 正常だが遅い | 帯域・バックエンド詰まり → DB・外部APIを疑う |
| DNS 解決不可 | DNS 障害（DDoS の可能性は残るが別系統） |
| ping 不通 | L3 不達（経路・機器・SG/NACL） |

### ここで気づけること

- curl が **即座に 200** を返すなら、その時点で **「サーバは生きている」** 確定 → ユーザ環境や CDN を疑う
- curl が **タイムアウト** しても、それは DDoS とは限らない（サーバ過負荷・FW 設定ミス・リソース枯渇）

## Phase 2：環境把握（1分）

### 2-1. OS とディストリ

```bash
uname -a
cat /etc/os-release       # Linux 各種
uname -v                  # FreeBSD なら "FreeBSD ..." と出る
```

ここで確定させるもの：
- RHEL 系（CentOS/Rocky/AlmaLinux）か
- Debian 系（Ubuntu/Debian）か
- FreeBSD か
- カーネルバージョン

→ 以降のコマンドパス・サービス名が変わる（本章末のディストリ差分表を参照）。

### 2-2. コンテナ内かどうか

```bash
# Docker コンテナ内なら存在する
ls -la /.dockerenv 2>/dev/null

# cgroup で確認
cat /proc/1/cgroup | head -3 | grep -E 'docker|containerd'

# systemd-detect-virt
systemd-detect-virt 2>/dev/null
```

コンテナ内だった場合の注意：
- ホスト側の `iptables` を直接叩けない
- ホストの `ss` と中の `ss` は違うものを見ている
- 攻撃がホスト側とコンテナ側のどちらに届いているかを見極める必要

### 2-3. クラウドかオンプレか

```bash
# AWS EC2
curl -s -m 2 http://169.254.169.254/latest/meta-data/ && echo "=> AWS EC2"

# dmidecode
dmidecode -s system-manufacturer 2>/dev/null
# "Amazon EC2" → AWS
# "VMware" / "QEMU" → 仮想基盤（多くはオンプレ）
```

→ AWS なら [06 の 6 章（AWS 対応）] の武器が使える。オンプレなら YAMAHA RTX / Cisco で止める。

## Phase 3：構成把握（2分）

「このサーバは何をしていて、誰と通信しているか」を把握する。配られた構成図は参考程度に。**実機から取った情報を信じる**。

### 3-1. 自ホストが提供するサービス

```bash
# リッスンしているポート
ss -tlnp 2>/dev/null
# FreeBSD なら
sockstat -4 -l
```

- 80/443 → Web
- 53 → DNS
- 25/587/465 → Mail
- 3306 → MySQL
- 5432 → PostgreSQL
- 8080/9000 → App サーバ（Tomcat/PHP-FPM）

### 3-2. このサーバが依存する他サーバ

```bash
# 外向きに張っている接続
ss -ntp 2>/dev/null | grep -vE '127\.|::1' | head -20

# リバースプロキシ設定
grep -RhE "proxy_pass|upstream|ProxyPass" /etc/nginx /etc/httpd /etc/apache2 2>/dev/null | head -20

# DB 接続先（アプリ側）
grep -RhE "DB_HOST|DATABASE_URL|mysql\.host" /var/www /etc /opt 2>/dev/null | head -10
```

### 3-3. 同セグメントのサーバ

```bash
arp -an
ip neigh show
```

### 3-4. ここで必ずホワイトボードに書く

**資料の構成図を見ずに、実機から取った情報だけで構成図を自作する。**

```
[外部]
  │
  ▼
[このホスト: nginx] 192.168.10.10
  │
  ├─▶ [Apache+PHP] 192.168.10.20 (Srv B)
  │          │
  │          ▼
  │   [MySQL] 192.168.10.30 (Srv C)
  │
  └─▶ [外部API] api.example.com (決済)
```

## Phase 4：ログ確認（2分）

### 4-1. アクセスログ（Web 層）

```bash
# RHEL 系
tail -n 200 /var/log/httpd/access_log

# Debian 系
tail -n 200 /var/log/apache2/access.log
tail -n 200 /var/log/nginx/access.log

# 直近 5 分のリクエスト数推移（秒単位）
awk '{print $4}' /var/log/httpd/access_log | cut -d: -f1-3 | uniq -c | tail -10
```

### 4-2. エラーログ

```bash
tail -n 100 /var/log/httpd/error_log       # RHEL
tail -n 100 /var/log/apache2/error.log     # Debian
tail -n 100 /var/log/nginx/error.log
```

見るポイント：
- `too many open files` → ファイルディスクリプタ枯渇
- `database connection failed` → **DB 側の問題**（DDoS じゃないかも）
- `SSL_accept error` → 証明書・TLS の問題
- `upstream timed out` → バックエンド詰まり（DDoS じゃないかも）

### 4-3. システムログ

```bash
# カーネル
dmesg | tail -30
dmesg | grep -iE "syn|flood|drop|out of|killed"

# systemd / journal
journalctl --since "10 min ago" --no-pager | tail -50
journalctl -u httpd -u apache2 -u nginx --since "5 min ago"
```

### 4-4. 判定表（ログの様子で何を疑うか）

| ログの様子 | 疑い | 次の一手 |
|---|---|---|
| access_log に大量リクエスト | L7 Flood 疑い | → [06] へ |
| access_log 静か・接続だけ多い | Slow HTTP 疑い | → [06 の 4-3] へ |
| error_log に DB 接続失敗連発 | **DB 側問題**（DDoS じゃない可能性） | DB のプロセスリスト確認 |
| error_log に upstream timeout | **バックエンド詰まり** | 後段サーバへ |
| SSL handshake 失敗連発 | 証明書・TLS 問題 | `openssl s_client` |
| dmesg に "SYN flood" | カーネルが検知 | ほぼ確定 → [06] へ |
| アクセス数は普段通り | DDoS ではない可能性大 | ユーザ側・経路・DNS |

## Phase 5：DDoS 判定に入る条件

以下の **2 つ以上** が揃ったら [06_DDoS対応詳細] へ進む。1 つだけでは早い。

- [ ] `access_log` / `ss -s` の接続数が通常の **10 倍以上**
- [ ] 送信元 IP が異常（海外・知らないレンジ・分散）
- [ ] `ss -s` の synrecv/estab/udp のいずれかが **しきい値超え**（数値は [06 の ⓪] 参照）
- [ ] CPU / 帯域 / ファイルディスクリプタのどれかが枯渇
- [ ] 攻撃特化シグネチャ（`dmesg` "SYN flood" / 同一 URL 連打 / ANY クエリ急増 等）

条件不足なら次章「DDoS じゃないケース」を先に潰す。

## DDoS じゃないケース 発生頻度ランク

現場で「遅い・見えない」の実際の原因は以下の確率で **DDoS 以外**。大会でも陽動やフェイント問題として出やすい。

| 順位 | 原因 | 見分け方 |
|---|---|---|
| **1** | **バックエンド DB 詰まり** | `SHOW PROCESSLIST`、slow query log、`pt-query-digest` |
| **2** | **外部 API の遅延**（決済・認証・検索） | アプリログの外部呼び出し timeout |
| **3** | **DNS 障害**（自社 or 上流） | `dig`、`/etc/resolv.conf`、`systemctl status named` |
| **4** | **証明書期限切れ / チェーン不正** | `openssl s_client -connect host:443 -servername host` |
| **5** | **ディスク満杯** | `df -h`、`/var/log/` が肥大 |
| **6** | **メモリリーク** | `ps aux --sort=-rss \| head`、`free -m` |
| **7** | **CDN / Cloudflare 障害** | CDN 側ステータス、オリジンへの直接アクセス |
| **8** | **スケジュールされたバッチ** | `cron`、`at`、深夜処理の残業 |
| **9** | **ネットワーク機器障害** | ルーター/スイッチログ、ポート状態 |
| **10** | **真の DDoS** | 上記を全部消去してから |

### 見分けコマンド集

```bash
# ① DB 詰まり
mysql -e 'SHOW FULL PROCESSLIST' | head -30
mysql -e 'SHOW GLOBAL STATUS LIKE "Threads_%"'

# ② 外部 API 遅延
# アプリログで外部呼び出し時間が長くなっているか
grep -iE "timeout|slow|took" /var/log/httpd/error_log | tail -20

# ③ DNS
dig @8.8.8.8 対象ホスト         # パブリック DNS で比較
systemctl status named          # 自社 DNS

# ④ 証明書
echo | openssl s_client -connect 対象:443 -servername 対象 2>/dev/null | openssl x509 -noout -dates

# ⑤ ディスク
df -h
du -sh /var/log/*

# ⑥ メモリ
free -m
ps aux --sort=-rss | head -10
```

## やってはいけない 10 項目（DDoS 判定前）

| # | NG 行動 | 理由 |
|---|---|---|
| 1 | curl で再現せずに「DDoS だ」と判定 | 環境側の問題を見逃す |
| 2 | 構成を把握せずに `iptables -j DROP` | 正規通信を巻き込む |
| 3 | ログを見ずにサーバ再起動 | 根本原因が消えて再発する |
| 4 | 審査員・顧客・大学 IP レンジを含む遮断 | 大会で大減点 |
| 5 | コンテナ内部から ホスト宛の攻撃を判断 | 見ているものが違う |
| 6 | `/etc/hosts` や DNS を疑わず即 DDoS 結論 | DNS 障害を DDoS と誤診しがち |
| 7 | access_log のパスを RHEL/Debian で間違える | 「ログに何もない」と誤認 |
| 8 | 時刻同期ズレのまま複数サーバのログ照合 | 時系列が壊れる |
| 9 | 単一サーバだけ見て判断 | 観測点・攻撃対象・被害箇所のズレを見落とす |
| 10 | 事実と事象（主観）を混同したまま技術判断 | 顧客の言い回しに引きずられる |

---

## 判定フロー（全体図）

```
クレーム受電
  │
  ▼ curl で再現する？
  ├─ 正常応答 ──────→ ユーザ環境側の問題（終了）
  └─ 異常（無応答 / 5xx）
       │
       ▼ 環境確認（OS / Docker / クラウド）
       │
       ▼ 構成確認（LB / バックエンド / 外部依存）
       │
       ▼ ログ確認（access / error / syslog）
       │
       ▼ 判定条件を 2 つ以上満たす？
       ├─ NO ──→ DDoS じゃないケース（上位10）を順番に潰す
       └─ YES ─→ [06_DDoS対応詳細] へ
```

## Linux ディストリ別コマンド差分

| 目的 | RHEL 系（CentOS/Rocky） | Debian 系（Ubuntu/Debian） |
|---|---|---|
| Apache サービス名 | `httpd` | `apache2` |
| Apache ログ | `/var/log/httpd/access_log` | `/var/log/apache2/access.log` |
| Apache 設定 | `/etc/httpd/conf/httpd.conf` | `/etc/apache2/apache2.conf` |
| モジュール有効化 | 設定ファイル内で `LoadModule` | `a2enmod / a2dismod` |
| サイト有効化 | `/etc/httpd/conf.d/` | `a2ensite / a2dissite` |
| nginx ログ | `/var/log/nginx/access.log` | 同じ |
| パッケージ管理 | `yum` / `dnf` | `apt` / `apt-get` |
| ファイアウォール標準 | `firewalld` (`firewall-cmd`) | `ufw` |
| iptables 永続化 | `/etc/sysconfig/iptables` / `iptables-save` | `iptables-persistent` パッケージ |
| MAC 機構 | SELinux | AppArmor |
| ログ | rsyslog + journald | journald 主体 |

`ss` `sysctl` `iptables` 本体のコマンドは共通。違うのは **サービス名・ログパス・永続化の仕方**。

## FreeBSD について

### 出る可能性

コンテストの公式機材リストに FreeBSD が含まれているため、**DNS/Mail サーバ役で出る可能性は残る**。ただし近年は Linux 主体。

| 観点 | 判断 |
|---|---|
| Web 攻撃対象として出る | 低（Linux が多い） |
| DNS/Mail で出る | 中（BIND/Postfix の古典的実装） |
| 準備配分 | Linux 80% / FreeBSD 20% |

### 差分（最低限）

| 目的 | Linux | FreeBSD |
|---|---|---|
| 接続状態 | `ss` | `netstat -an` / `sockstat` |
| パケットフィルタ | `iptables` / `nft` | `pf` / `ipfw` |
| サービス起動 | `systemctl` | `service` |
| インターフェース名 | `eth0` / `ens5` | `em0` / `bge0` / `igb0` |
| カーネルパラメータ | `net.ipv4.*` | `net.inet.*` |
| Apache パス（ports） | `/etc/httpd/` | `/usr/local/etc/apache24/` |
| ログ | `/var/log/httpd/` | `/var/log/` 直下 |

FreeBSD の具体的コマンドは [06_DDoS対応詳細](./06_DDoS対応詳細.md) の各節に併記あり。
