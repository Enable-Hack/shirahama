# 05. 全体チェックシート

> インシデント発生の第一報から完了報告まで、**抜け・漏れを防ぐ** ためのチェックシート。
> 1件のインシデントにつき 1枚印刷 or 1ファイルコピーで使用。

---

## 事案ヘッダ（最初に埋める）

```
事案ID：                    （例：INC-20260420-001）
発生検知：   mm/dd HH:MM    検知経路：□ユーザ電話 □顧客担当 □経営層 □外部通報 □監視アラート
事案分類：   □フィッシング □Web改ざん □ランサム □DDoS □マルウェア □情報漏えい □不正送信 □不明
優先度：     □P1 □P2 □P3
対応チーム：
　電話応対：           記録：             技術調査：            指揮：
```

---

## Phase 0：受電・第一報（0〜10分）

```
[ ] 電話応対係が [01_受電ヒアリングシート] を使い聞き取り
[ ] 冒頭3項目（所属／氏名／折り返し連絡先）を取得
[ ] パターン（A/B/C/D）を判定
[ ] 受電時刻を記録
[ ] 相手に「端末に触らない／電源を切らない」を依頼
[ ] 折り返し時刻を相手に約束（例：10分以内）
[ ] 記録係がホワイトボードに時系列で記録開始
[ ] 受電ヒアリングシートを技術係・指揮係にも共有
```

### Phase 0 完了条件
- 誰が・何を・いつ起きたか の骨子がチームで共有されている
- 相手への **折り返し約束時刻** が決まっている

---

## Phase 1：一次判定（10〜30分）

```
[ ] [02_技術担当引き継ぎテンプレート] を使い、事実と申告を分けて整理
[ ] 緊急度・優先度を判定（1人で決めず、指揮係＋技術1名で合意）
[ ] 対象システム・対象セグメント・対象ユーザを特定
[ ] 技術担当に依頼事項を明記（抽象指示はNG）
[ ] 顧客担当者への一次連絡（完了していない旨と、次回報告時刻）
[ ] チーム内の次回レビュー時刻を決定（15〜30分後）
```

### 一次連絡テンプレ（Phase 1 で相手に返す）

```
ご連絡いただいた件、現在以下のように対応を開始しています。
　・受電時刻：〇〇
　・現在の状況：〇〇
　・次のアクション：〇〇
　・次回ご連絡：mm/dd HH:MM まで

ご不明点があれば、この番号にご連絡ください。
```

### Phase 1 完了条件
- 緊急度の判定がついている
- 技術担当に **具体的な** 調査依頼が渡っている
- 相手に一次連絡が済んでいる

---

## Phase 2：封じ込め（拡大停止）

> **復旧よりも先にやる**。拡大を止めないと被害が増える。

```
[ ] シナリオ別の封じ込め手順を [03_シナリオ別対応プレイブック] で確認
[ ] 封じ込めの実施前に指揮係の承認を取得
[ ] 封じ込め実施時刻・実施内容を記録
[ ] 封じ込め後に効果を確認（拡大が止まったか）
[ ] 顧客担当者に封じ込め完了を中間報告
[ ] バックアップ媒体・ログの保全（上書き防止）
```

### 封じ込めの例（シナリオ別）

| シナリオ | 典型的な封じ込め |
|---|---|
| フィッシング | 該当アカウントのパスワードリセット、セッション切断 |
| Web改ざん | サイトをメンテナンスモード／公開停止 |
| ランサム | 該当端末LAN切離、ファイルサーバを読み取り専用 |
| DDoS | UTM/FWで攻撃元IP遮断 |
| マルウェア | 該当端末LAN切離、C2通信先を遮断 |
| 情報漏えい | 共有リンクの無効化、メール回収、ファイル削除 |
| 不正送信 | 該当アカウント無効化、SMTPリレー閉鎖 |

### Phase 2 完了条件
- 拡大が停止している（観測で確認）
- 封じ込めの内容と時刻が記録されている
- 顧客に封じ込め済みの旨を伝えてある

---

## Phase 3：調査・原因特定

```
[ ] 侵入経路・原因の仮説を立てる（複数候補でOK）
[ ] ログ・証跡を収集（時刻・ホスト・取得者を明記）
[ ] 仮説を1つずつ検証（裏取り前の断定は禁止）
[ ] 他システム・他ユーザへの影響有無を確認
[ ] 陽動の可能性（同時刻の別インシデント）を確認
[ ] 調査結果を [02] の「調査結果記入欄」に記録
[ ] 原因が特定できたら、指揮係に報告して復旧判断へ
```

### 調査で必ず集めるログ（最低限）

- 認証ログ（AD、SSH、アプリログイン）
- アクセスログ（Web、ファイル共有、DB）
- ネットワーク通信ログ（FW／UTM、L3スイッチ）
- メールログ（送信・受信）
- プロセス・サービス起動ログ
- 監視システムのアラート履歴

### Phase 3 完了条件
- 直接原因が特定できている（または「ここまで分かった／ここからは不明」が整理されている）
- 影響範囲の確定（利用者・データ・システム単位）
- 復旧方針が決まっている

---

## Phase 4：復旧

```
[ ] 復旧方針を指揮係が承認
[ ] 復旧前に封じ込めが維持されていることを再確認
[ ] バックアップから戻す場合は、戻す前に脆弱性対応を済ませる
[ ] 復旧手順を記録（いつ／誰が／何を）
[ ] 復旧後、正常稼働を確認（サービス動作、ログ健全性）
[ ] 監視を強化（再感染・再攻撃に備えて）
[ ] 顧客担当者に復旧完了を伝達（完全復旧なのか一次対応完了なのか明示）
```

### 復旧で失敗しやすいパターン

- 経路を塞がずに復元 → 即再侵害
- 「動いたのでOK」で監視を下げる → 見落とし
- 暗黙の手順で復元 → 他メンバーが再現できない
- バックアップ媒体が感染 → 復元で再感染

### Phase 4 完了条件
- サービスが正常に稼働している
- 復旧作業の記録が残っている
- 強化監視が動いている

---

## Phase 5：完了報告

```
[ ] [04_完了報告テンプレート] の完了判定チェックリストを確認
[ ] 相手別に適切なテンプレを選ぶ（経営層／担当者／ユーザ／外部通報者）
[ ] 時系列・影響範囲・原因・今後 を明記
[ ] 電話連絡後、正式記録としてメールを送る（必要な場合）
[ ] 連絡時刻を記録
[ ] 未解決の継続タスクを明示（完了と未完了を混同しない）
```

### Phase 5 完了条件
- 対象全員（経営層／担当者／ユーザ）への連絡が済んでいる
- 完了なのか一次対応完了なのかが相手に明確に伝わっている
- 記録が残っている（後日の問い合わせ対応が可能）

---

## Phase 6：振り返り（対応終了後）

```
[ ] 時系列ログを整理（対応記録ドキュメント化）
[ ] KPT でチーム振り返り
　  K(keep)：よかった点
　  P(problem)：悪かった点・課題
　  T(try)：次回に試すこと
[ ] 再発防止の技術タスクをリスト化
[ ] 対応マニュアル・プレイブックの改善点を記録
[ ] 顧客向け最終報告書の作成（必要な場合）
```

---

## 並行対応・複数インシデントの管理

> 大会では **同時に複数のインシデント** が発生することが多い。陽動にも注意。

```
[ ] 事案ごとに別IDを振る（INC-YYYYMMDD-001, 002, …）
[ ] 事案ごとに担当（主担当）を決め、混同を避ける
[ ] ホワイトボードを事案ID単位で区切る
[ ] 指揮係は全事案を把握、**優先度の並び替え** を継続
[ ] 陽動の可能性を常に念頭に（1件に集中しすぎない）
[ ] 電話が複数同時に来たら、2件目は折り返し or 記録係に引き継ぎ
```

```

mm/dd HH:MM  【中間】顧客担当者に封じ込め完了連絡
mm/dd HH:MM  【調査】セッション全切断、他端末への影響確認

```

## チーム役割ごとのチェック

### 電話応対係
```
[ ] 冒頭の3項目を確実に取る（所属／氏名／折り返し）
[ ] 相手を遮らず最後まで聞く
[ ] 復唱して内容確認
[ ] 次回連絡時刻を約束
[ ] 原因断定・安心づけをしない
[ ] 自分で判断せず、指揮係に上げる
```

### 記録係
```
[ ] 時系列で記録（mm/dd HH:MM を必ず）
[ ] 事実と申告を分けて記録
[ ] 判断とその根拠を記録
[ ] 記録を全員が見える場所に掲示
[ ] 1事案=1シートを徹底
```

### 技術調査係
```
[ ] 依頼内容を明確に受け取る（不明なら聞き返す）
[ ] ログの取得時刻を記録
[ ] 事実と推定を分けて報告
[ ] 勝手に復旧操作を始めない（指揮承認必須）
[ ] 調査結果を記入欄に戻す
[ ] 追加調査が必要なら要求する
```

### 指揮・意思決定係
```
[ ] 全事案を俯瞰し、優先度を管理
[ ] 判断は「根拠＋決定＋時刻」の3点で記録
[ ] 顧客対応の窓口を一本化
[ ] 復旧・封じ込めの実施を明確に承認
[ ] 定時レビューを回す（15〜30分ごと）
[ ] 対応終了時にチェックリスト全項目を確認
```

---

## よくある抜け・漏れ（大会頻出）

- 受電時に **折り返し先を聞き忘れる**
- 時刻を **「さっき」** で済ませてしまい後で再現できない
- 技術担当が **勝手に再起動** して揮発データを消失
- 封じ込め前に復旧を始めて **再感染**
- 陽動を見抜けず、**1件に全員集中している間に2件目が来る**
- 完了報告で「**大丈夫です**」と断定し、後で覆される
- 顧客担当者と経営層に **別々の話** を伝えてしまい齟齬
- 記録が個人メモに散らばり、**チームで共有できない**

---

## 最終確認（事案クローズ前）

```
[ ] 全 Phase のチェック項目が埋まっている
[ ] 対応記録が1箇所にまとまっている
[ ] 顧客・ユーザ・関係者への連絡が済んでいる
[ ] 完了と未完了が明確に分かれている
[ ] 再発防止策・継続タスクがリスト化されている
[ ] チーム振り返りの予定 or 完了
```
