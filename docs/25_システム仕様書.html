白浜 AI ログ分析パイプライン
システム仕様書

電話受電起点のインシデント対応を Claude Code の slash command で半自動化する。 本書は 5 つのコマンド (2026-05-04 簡素化第二弾後) の仕様、4 層アーキテクチャの内部、典型 4 シナリオでの操作手順を、 コードと実出力ベースで記述した運用者向けリファレンス。

§ 1はじめに

1.1このドキュメントについて#

本書は 運用者 (技術調査係 / 指揮係) が実際の試合中に参照するための仕様書。設計思想や 30 秒ピッチは 21_システム解説.md を参照。本書はそれより 1 段詳しく、「このコマンドを叩くと何が走り、何が出力され、次に何をすべきか」 を 1 ページで答える形にしてある。

各コマンド節は (a) 仕様 → (b) 内部処理 → (c) 出力形式 → (d) 次の手 の 4 ブロック構成。§4 のワークドエグザンプルで実際の通報シナリオを最初から最後まで追跡する。

2.13 層 + 議論ループ + 物理ゲート#

2.3設計判断#

Q1. なぜ 60 項目を全部 pattern 化しなかったか

過去環境固有の CVE (rainloop 1.12.0 等) に過学習すると、本番で別 CVE が出たとき検知ゼロになる。6 カテゴリに圧縮した汎用 pattern で「rainloop でも roundcube でも捕捉する」形にした。

Q2. なぜ Mock + Claude の二段か

Mock 単独では未知攻撃を取りこぼす。Claude 単独では API コストとレイテンシで 30 分制約を破る。Mock filter で 8 割減 → grey だけ Claude で速度とコストを両立。

Q3. なぜ deny / ask / allow の 3 層か

「絶対実行しない」「人間判断要」「即実行」の 3 段階で粒度を分けることで、安全と速度を両立。read-only な dig / tail を毎回確認していては時間が足りない。一方で iptables -j DROP を allow にしては事故る。

Q4. なぜ未分類ログも Claude に送るのか

analyzer の 41 pattern は 過去環境で見えた攻撃のスナップショット。本番では新しいログ行が出る可能性が高い。「マッチしなかった行」 を捨てると未知攻撃を取りこぼす。一方、全行を Claude に送るとコストとレイテンシが破綻する。妥協点が filter_known_good_logs() — 配布アカウントと自 IP の行だけ機械的に drop し、残った「謎のログ」を Claude に意味推定させる。

§2.5. ⚠ タイムゾーン (致命的) ─ サーバが UTC でも JST でも対応

大原則: ユーザ入力は常に JST (受電シート / 時間窓引数 / hearing JSON すべて JST)。サーバ側 (Rocky / FreeBSD) は UTC でも JST でもあり得るが、/incident が ssh で date +%z を叩いて自動検出 → 入力 JST を各サーバ TZ に変換してからログ検索する。
  • OCI Rocky (デモ) = +0000 (UTC、確認済)
  • 本番 victor (Rocky) = ? (大会開始まで未確認、UTC の可能性高い)
  • 本番 bravo (FreeBSD) = ? (同上、cloud image なら UTC が多い)
事故シナリオ: ユーザ「12:00 JST 頃から」と聞く → /incident 12:00-12:30 を打つ → サーバが UTC なら本来は UTC 03:00-03:30 のログを見る必要がある。修正前は時刻文字列マッチで偶然 UTC 12:00 のログ (= JST 21:00) を取ってきて 9 時間ズレた別の時間帯のログ取得になっていた。2026-05-05 修正済。

確認方法: /incident の出力冒頭に「victor TZ: +0000 → ローカル時刻 03:00〜03:30」のように変換結果が出る。これがおかしかったら時刻入力を見直す。

受電 JSON: hearing__*.json の time_window_jst = ユーザ入力 (JST 表記) / time_window_utc = JS 側で自動換算 (UTC 表記)。両方 JSON に入る。
