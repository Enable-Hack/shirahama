白浜 AI ログ分析パイプライン
システム仕様書

電話受電起点のインシデント対応を Claude Code の slash command で半自動化する。 本書は 8 つのコマンドの仕様、4 層アーキテクチャの内部、典型 4 シナリオでの操作手順を、 コードと実出力ベースで記述した運用者向けリファレンス。

§ 1はじめに

1.1このドキュメントについて#

本書は 運用者 (技術調査係 / 指揮係) が実際の試合中に参照するための仕様書。設計思想や 30 秒ピッチは 21_システム解説.md を参照。本書はそれより 1 段詳しく、「このコマンドを叩くと何が走り、何が出力され、次に何をすべきか」 を 1 ページで答える形にしてある。

各コマンド節は (a) 仕様 → (b) 内部処理 → (c) 出力形式 → (d) 次の手 の 4 ブロック構成。§4 のワークドエグザンプルで実際の通報シナリオを最初から最後まで追跡する。

2.3設計判断#

Q1. なぜ 60 項目を全部 pattern 化しなかったか

過去環境固有の CVE (rainloop 1.12.0 等) に過学習すると、本番で別 CVE が出たとき検知ゼロになる。6 カテゴリに圧縮した汎用 pattern で「rainloop でも roundcube でも捕捉する」形にした。

Q2. なぜ Mock + Claude の二段か

Mock 単独では未知攻撃を取りこぼす。Claude 単独では API コストとレイテンシで 30 分制約を破る。Mock filter で 8 割減 → grey だけ Claude で速度とコストを両立。

Q3. なぜ deny / ask / allow の 3 層か

「絶対実行しない」「人間判断要」「即実行」の 3 段階で粒度を分けることで、安全と速度を両立。read-only な dig / tail を毎回確認していては時間が足りない。一方で iptables -j DROP を allow にしては事故る。

Q4. なぜ未分類ログも Claude に送るのか

analyzer の 41 pattern は 過去環境で見えた攻撃のスナップショット。本番では新しいログ行が出る可能性が高い。「マッチしなかった行」 を捨てると未知攻撃を取りこぼす。一方、全行を Claude に送るとコストとレイテンシが破綻する。妥協点が filter_known_good_logs() — 配布アカウントと自 IP の行だけ機械的に drop し、残った「謎のログ」を Claude に意味推定させる。

    > /tmp/incident_secure.log

3.1.5b 未分類ログの 3 段ふるい (2026-05-03 強化)

analyzer の 41 pattern にマッチしなかったログ行は run_with_unmatched() で別バケツに分けられ、そのまま Claude に投げると 1000 件超でトークン爆発するため mock_backend.filter_known_good_logs() で 3 段ふるいに掛ける。実測: raw 1896 件 → 63 件 (97% 削減)。

Layer 2システムノイズ NOISE_PATTERNS に該当 (pam_unix session / sshd preauth disconnect / systemd-user session / named dumping master file / sendmail unqualified host name 等 8 個)drop (新規)

3.1.5c タイムスタンプ正規化 (2026-05-03 追加)

未分類ログの dict に ISO 8601 UTC 形式の ts フィールドを付与する (analyzer 側で _parse_syslog_ts() ヘルパが May 3 08:19:20 → 2026-05-03T08:19:20+00:00 に変換)。Claude が時系列推論する際に raw 文字列を再パースする必要がなくなる。

# 旧: タイムスタンプは raw 文字列に埋まったまま
{'raw': 'May  3 08:19:20 bravo named[...]: ...', 'type': 'named'}

# 新: 構造化された ts フィールド付き
{'raw': 'May  3 08:19:20 bravo named[...]: ...',
 'type': 'named',
 'ts':   '2026-05-03T08:19:20+00:00'}

3.1.8 既知の制限

  • analyzer の webapp/auth-bruteforce 集約が wp-login.php POST × 20 を拾えていないケースが報告されている。§4.A 参照。
  • §2 の WINDOW="$1" が ad-hoc bash 経由だと位置引数を受け取れず、Claude が手動で WINDOW="..." 置換する必要あり。
  • ログタイムスタンプは UTC 前提。JST から打つ場合は 必ず -9h 変換。
  • analyze_maillog / analyze_named / analyze_secure で signals.evidence.ip が未設定 → attacker_ips cross-ref (§3.1.5b Layer 3b) が効きにくい。現状は 10.1.129.0/24 経由の Layer 3c が主力。analyzer 改修は別タスク。

  • build mode (rocky) のオペレータ自身の sudo COMMAND= に既知侵害 IP を grep した行が forensic として残る (false positive)。production mode の manage 運用では発生しない。
