インシデント対応 スイムレーン

一般インシデント（DDoS・侵害・情報漏えい等、種別問わず）の 初動 → 中間 → 終了 の流れを 4レーン並行（電話対応・指揮・サーバー・ネットワーク）で時系列に可視化。 電話が鳴った直後から完了報告までの各レーンの動きを、01／02／02.1／03／04／05／06 を横串で通した1枚に集約。

原則1：観測点 ≠ 攻撃対象 ≠ 被害箇所 症状が見えた場所、パケットが来ている場所、枯渇している場所は別物。3つを分けて考える。
原則2：事実と事象を分ける 「遅い」「見えない」は主観。時刻・URL・IP・数字で記録する。
原則3：連絡線を一本化 顧客への回答窓口は電話対応係のみ。技術担当から直接返答させない。
原則4：封じ込め > 復旧 経路を塞ぐ前に復旧しない。復旧 → 再発の玉突きを避ける。
初動 0〜15分 ▶ 中間 15〜60分 ▶ 終了 60分〜 ※ 時間は目安。事案規模で伸縮する。
初動
受電 → 切り分け → 方針決定（0〜15分）
中間
封じ込め → 調査 → 対応実施（15〜60分）
終了
復旧確認 → 報告 → クローズ（60分〜）
電話対応
Phase 0 受電 01
  1. 共通冒頭（「お電話ありがとうございます」）
  2. 冒頭3項目：所属・氏名・折り返し先を復唱して確保
  3. パターン判定：A苦情 / B報告 / C経営層 / D外部通報
  4. 相手に「触らない・電源切らない・再起動しない」を依頼
  5. 折り返し時刻を約束（10〜15分以内）
  6. 受電シートを指揮・記録係に即共有
NG：原因断定・安心づけ（「ウイルスです」「大丈夫です」）を言わない。
→ 01_受電ヒアリングシート
Phase 1-2 中間報告 04
  1. 15〜30分ごとに顧客担当へ中間連絡（進捗・見通し）
  2. 経営層からの電話応対（事実・対応中・見通し・判断事項の4点）
  3. 相手からの追加情報（新症状・目撃）を取り込み指揮へ
  4. 他ユーザへの広報要否を指揮に確認
  5. 2件目以降の受電（陽動の可能性を意識して受ける）
  6. 連絡時刻・相手・内容は1件1行で記録
連動：指揮の「報告頻度合意」に従い発信。技術者を直接つながない。
Phase 3-5 完了報告 04
  1. 完了判定チェックリストを指揮と確認
  2. 相手別にテンプレ選択：経営層 / 担当 / ユーザ / 外部
  3. 「完了」と「一次対応完了」を明確に区別して伝える
  4. 電話後、メールで裏付け記録を送付
  5. 次回連絡時刻（継続対応があれば）を約束
  6. 連絡ログを1箇所にまとめ指揮へ提出
NG：「もう絶対に大丈夫」「完全に分かりました」と断定しない。
指揮
Phase 0-1 統制開始 02 / 05
  1. チーム招集（4人全員が事案を把握）
  2. 事案ID採番（例 INC-20260420-001）
  3. ホワイトボードで時系列記録開始
  4. 02 引き継ぎテンプレで事実と申告を整理
  5. 緊急度（高/中/低）・優先度（P1/P2/P3）判定
  6. 各レーンへ「何を・いつまでに・どこまで」の具体依頼
  7. 中間報告タイミング合意（例 15分後）
→ 02_引き継ぎテンプレート / 05_全体チェックシート
Phase 2-3 判断 03 / 06
  1. 両技術レーンの仮説を統合しシナリオ特定
  2. 03 プレイブックの該当シナリオを技術へ渡す
  3. 封じ込め実施を承認（範囲・タイミング）
  4. 陽動の可能性を常時監視（別ログ並行確認）
  5. 複数インシデント管理（優先度の再評価）
  6. 定時レビュー（30分経過時点で一度集合）
  7. 完了見通し時刻を各レーンから集約
連動：封じ込めは指揮承認が必須。技術の独断で止めない。
Phase 4-5 クローズ 05
  1. 05 全体チェックシートで完了判定を確認
  2. 継続タスク（再発防止・監視強化）を整理
  3. 事案クローズ宣言 or「一次対応完了」宣言を明確に
  4. 対応記録を1箇所にまとめ（時系列・判断・根拠）
  5. KPT振り返り（時間があれば実施）
  6. 体制戻し（次事案に備えてローテーション）
NG：完了判定7項目（05参照）が埋まる前に「完了」と言わない。
サーバー
Phase 0-1 切り分け 02.1
  1. 再現確認：curl / dig / ping / nc で失敗レイヤ特定
  2. 環境把握：OS・ディストリ・Docker・クラウド判別
  3. 構成把握：自ホストの役割と他サーバ依存関係
  4. ログ確認：access / error / syslog（距離を取って読む）
  5. 対応前にログ・画面のスクリーンショット保全
  6. 事実 vs 相手申告を分けて仮説を立てる
  7. 仮説（DDoS疑い/侵害疑い/別原因）を指揮へ報告
NG：再現確認なしで iptables DROP を打たない。
→ 02.1_先にやること
Phase 2-3 封じ込め・調査 03 / 06
  1. 指揮承認後に封じ込め実施（アカウントロック・サービス停止・iptables等）
  2. 詳細調査：侵入痕跡・不審プロセス・改ざん痕跡
  3. 認証ログ監査・アクセスログ深掘り
  4. バックアップ状態確認（復旧可否）
  5. 復旧方針を指揮へ提示（完全復旧/一部停止/代替）
  6. 調査結果を02引き継ぎテンプレの結果欄へ記入
  7. 陽動の痕跡（別ホストのログ異変）も並行監視
連動：NWと時系列を同期（ログ時刻のズレに注意）。
Phase 4-5 復旧・記録 03 / 04
  1. 指揮承認のもと復旧作業実施
  2. 動作確認（正常稼働・ログ健全性）
  3. 監視強化設定（再侵害・再発の早期検知）
  4. 時系列ログ整理（報告書に使える形式）
  5. 残タスク（未解決項目）を指揮へ申告
  6. 対応内容を04完了報告の「実施した対応」へ渡す
NG：経路を塞がずに復元しない（即再感染）。
ネット
ワーク
Phase 0-1 観測 02.1
  1. 対象セグメント・VLAN を特定
  2. 機器状態：YAMAHA `show environment` / Cisco `show processes cpu`
  3. 全体トラフィック確認（通常比較）
  4. 外部到達性・他セグメントとの疎通
  5. 機器ログ取得（`show log`、syslog）
  6. 配布構成図と実機のズレ確認（実機優先）
  7. 仮説（攻撃/通常増/別原因）を指揮へ報告
NG：構成図を鵜呑みにしない。実機から取った情報で自作する。
Phase 2-3 封じ込め 03 / 06
  1. 投入前に `show config` → 既存フィルタ番号をメモ
  2. ロールバックコマンドを先に用意してから投入
  3. YAMAHA: `ip filter` 追加（既存番号も含めて適用）
  4. Cisco: 名前付きACL で deny 追加
  5. AWS: NACL / SG / WAF Rate-Based Rule
  6. 投入後に効果確認（`show status ip filter` / CloudWatch）
  7. 他セグメント拡大・陽動トラフィックを並行監視
連動：サーバと同じ時刻基準でログを残す（NTP 確認）。
Phase 4-5 棚卸し 05
  1. 復旧後の接続確認（顧客側・他セグメント）
  2. 投入フィルタの棚卸し（不要なら外す / 残すものは記録）
  3. 監視アラート設定の継続要否を指揮と合意
  4. 構成変更の記録（いつ・誰が・何を）
  5. ロールバック可能状態を確認
  6. 構成図・運用ドキュメントの更新提案
NG：投入したルールを記録せず「終わり」にしない。

★ 重要判断ポイント（いずれも指揮が最終決定）

DP1：受電直後（0〜5分）DDoSと決め打ちせず、02.1の切り分け手順を踏むか
DP2：切り分け後（10〜15分）インシデント種別の仮説確定。03のどのシナリオに進むか
DP3：封じ込め実施前（15〜25分）サービス停止を伴うか。顧客への事前連絡が必要か
DP4：中間時点（30分）陽動の可能性を再評価。他ログ異変の有無を確認
DP5：復旧前経路（侵入経路・攻撃経路）が塞がっているか再確認
DP6：完了宣言前「完了」か「一次対応完了」か。継続監視が必要か

全レーン共通 NG 行動

  • 事実と事象（主観）を混同したまま技術判断
  • 受電時に折り返し先を聞かずに切る
  • 時刻を「さっき」「いま」で済ませる
  • 技術担当が顧客に直接返答（連絡線の二重化）
  • 封じ込め前に復旧開始して再発させる
  • 単一サーバだけ見て横断確認を怠る
  • 陽動の可能性を忘れて1件に全員集中
  • 審査員・大学側IPレンジを含む遮断
  • 投入コマンドにロールバック手順を用意しない
  • 「止まった」=「防げた」と誤認して完了宣言
