§ 04 — Architecture

4 層 ＋
物理ゲート

観測 (analyzer) → 判断 (Mock + Claude) → 検証 (validator) → 提案 → settings.json 物理ゲート。 LLM が暴走しても実機に届かない設計。

/var/log/*   (httpd / secure / maillog / named)

Rule-based

LLM

Hard-fail
×

物理ゲート — settings.json

deny:67 / ask:37 / allow:64

Physical Gate
→ 実機 (victor 10.1.1.2 / bravo 10.1.1.1)

分離の理由

  • 観測層 「動き続ける」が責務。ルールベースで LLM 不要、可用性の下支え。
  • 判断層 1 配布アカウント / 自チーム IP を物理 drop し、Claude API への流量を ~80% 削減。
  • 判断層 2 「質を上げる」が責務。grey signal だけ集約し、顧客向け文面まで生成。
  • 検証層 「暴走を止める」が責務。LLM 出力の壊れた regex / ReDoS を all-or-nothing 棄却。
  • ゲート nsupdate / systemctl stop / Edit(/etc/**) は実行段階で物理ブロック。

§ 06 — Yellow Line

やらないこと

本番までに着手しないこと、AI に任せないこと。「触らない哲学」を機能の不在として宣言する。

CAUTION · DO NOT CROSS
  • Never

    AI が顧客に直接電話 / メール送信

    採点軸「顧客対応力」は人間の温度感調整が本質。AI は文面生成までで停止する。

  • Never

    AI が破壊系コマンドを自動実行

    nsupdate / systemctl stop / Edit(/etc/**) は settings.json で物理ブロック済。Claude が善意で提示しても通らない。

  • Never

    AI が PukiWiki に直接投稿

    認証情報を Claude に渡さない方針。/ticket はマークアップ生成までで、投稿は人間が責任を持つ。

  • Never

    AI が複数インシデントの優先度を決める

    優先度判定は指揮係の領域。AI が割り込むと混乱する。

  • Never

    動的な攻撃カテゴリ再判定

    1 件ごとに /incident を再起動すれば足りる。動的リ分類は実装複雑度に見合わない。

  • Never

    未知攻撃カテゴリの新規 pattern 動的生成

    6 カテゴリ汎用化の設計判断と矛盾。本番は既存 pattern で押し、未知は人間判断に倒す。
