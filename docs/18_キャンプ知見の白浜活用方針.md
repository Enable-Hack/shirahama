## 0. なぜキャンプの知見が白浜に効くのか（30 行）

キャンプで学んだ核は技術ではなく **「攻撃を守りに変換する 4 層パイプライン」** という考え方:

```
ログ → ① 観測層 → ② 判断層 → ③ 検証層 → ④ 提案層 → 人間レビュー → 反映
        ルール検出   LLM/Mock    壊れた提案棄却  WAF/iptables 設定生成
```

これは Web に限らず **「観測されたシグナルから防御アクションへ変換する」全プロセスに適用可能**。
白浜の評価軸 4 つ全部に対応する:

| 白浜の評価軸 | キャンプ知見でカバーできる部分 |
|---|---|
| 技術対応力 | 観測層（パターン検出）+ 判断層（攻撃か正常か） |
| 顧客対応 | Claude による「**やさしい説明文**」自動生成（後述 §6） |
| 報告・連絡 | `explain_to_operator_ja()` 形式の「現状把握→推奨アクション」テンプレ |
| 報告書の質 | 4 層分離による **「やったこと」「判断理由」「やらなかったこと」** の構造化 |
| プロセス遵守 | 検証層 + 人間承認ゲート = 「改修前の承認」プロセスそのもの |

**核心**: 「速い防御は粗い、賢い防御は遅い、だから組み合わせる」(教材 p.7)
→ 白浜では **「速い人間判断（playbook）+ 賢い AI 整理（Claude）+ 確実な実機反映（手動承認）」** の 3 段運用に変換する。

## 1. 全体方針 — キャンプの 4 層を白浜にどう拡張するか

### 1.1 観測層 (Analyzer)

キャンプ版は nginx の access.log 1 本だったが、白浜では **複数ログソース**を観測対象にする:

| ログ | 対象攻撃 | パース難易度 |
|---|---|---|
| `/var/log/nginx/access.log` (victor) | XSS / SQLi / RCE / Web スキャン | 低（JSON にできるはず） |
| `/var/log/secure` (Linux) | SSH brute force / sudo 不正 | 中（行ベース） |
| `/var/log/auth.log` 同等 (FreeBSD) | bravo の SSH | 中 |
| `/var/log/maillog` (postfix/dovecot) | フィッシング転送 / メール bomb / 認証失敗 | 中 |
| `/var/log/messages` | カーネル / DDoS 兆候 / OOM | 高（雑多） |
| `named` ログ (BIND) | nsupdate / AXFR / DDoS amplification | 中 |
| ヤマハ syslog (RTX1200/1210) | NW 層 DDoS / ACL 違反 | 中 |
| `wtmp` / `last` | ログイン履歴（obuchi 深夜接続みたいな兆候） | 低 |

→ **キャンプの `analyzer.py` をフォークして、pattern_tag を白浜用に拡張**する。
   後述 §3 で各攻撃ごとのシグネチャを定義。

### 1.2 判断層 (Judgment)

| バックエンド | 白浜での使い所 |
|---|---|
| **Mock**（決定論的・即時） | 既知パターン（rainloop CVE、PHP 7.2、pkexec）の **初動 5 分の自動判定** |
| **Claude**（API・2-5秒） | ノイズの多いシグナル束を **「報告書 1 段落」に集約**、未知攻撃の意味推定 |
| Swallow（ローカル LLM） | 使わない（CPU 推論で 1-3 分は競技で遅すぎる） |

**運用ルール**:
- **遮断アクション**は Mock で即決（人間が事前に承認したルールセットのみ）
- **報告・状況整理・優先順位付け**は Claude を回す
- Claude は判断を**ブラックボックスにしない**ため、必ず `rationale_ja` を出させて報告書に貼る

### 1.3 検証層 (Validator)

キャンプの validator (`rule_id` 形式 / action allowlist / ReDoS 検出 / all-or-nothing) は **そのまま流用**。
ただし白浜では生成対象が WAF だけでなく **iptables / pf / named.conf / Cisco ACL** に広がるので、レンダラー側で対応する。

→ **「LLM が壊れたコマンドを生成しても実機に流れない」セーフティネット**は競技で必須。
   `iptables -F` を AI が出してしまうような事故を物理的に防ぐ。

### 1.4 提案層 (Renderer)

| 出力先 | キャンプ実装 | 白浜での拡張 |
|---|---|---|
| nginx WAF ルール | 実装済み（`if ($args ~ ...) { set $waf_block 1; }`） | victor の Apache/nginx に流用可 |
| iptables / pf | 無し | **追加実装が必要**（後述 §3 で雛形） |
| named.conf (BIND) | 無し | `allow-update { none; }` パッチ生成 |
| 顧客向け説明文 | 無し | **追加実装**（Claude が日本語で生成） |
| 経営層報告書 | `explain_to_operator_ja()` あり | そのまま流用 |

## 2. 攻撃カテゴリ別の方針マトリクス

| カテゴリ | 観測 | 判断 | 反映 | 顧客対応 |
|---|---|---|---|---|
| Web (XSS/SQLi/RCE) | nginx log | Mock 即決 | WAF ルール | 「サイト一時停止のお知らせ」 |
| DDoS | NW syslog + ss | Claude で帯域パターン分析 | iptables/RTX ACL | 「アクセスしづらい状況のご報告」 |
| フィッシング | maillog + 通報 | Claude で本文判定 | postfix reject + 周知 | 「不審メール注意喚起」 |
| DNS 改ざん | named log + dig | Mock（パターン明確） | named.conf 修正 | 「DNS 復旧のご報告」 |
| SSH brute / 既知CVE | secure + auditd | Mock + Claude 文脈 | iptables + パッチ | 「不正ログイン試行検知」 |

| 内部不正 / 横展開 | wtmp + sudo log | Claude 必須 | 個別対応 | 慎重に（誤認の影響大） |

## 3. カテゴリ別詳細

### 3.1 Web 攻撃（キャンプ直結・最強カバー領域）

**観測対象**: victor の `/var/log/nginx/access.log` または `/var/log/httpd/access_log`

**キャンプから直接持ち込めるシグネチャ**（pattern_tag を analyzer に追加）:

```python
WEB_PATTERNS = {
    'xss/script-tag':      r'(?i)<script',
    'xss/onerror-handler': r'(?i)onerror\s*=',          # ← Day1 で WAF バイパスした手口
    'xss/javascript-uri':  r'(?i)javascript:',
    'sqli/union-select':   r'(?i)union\s+select',
    'sqli/or-tautology':   r"(?i)or\s+['\"]?1['\"]?\s*=\s*['\"]?1",  # 引用符ありもカバー
    'rce/system-call':     r'(?i)(system|exec|passthru|shell_exec)\s*\(',
    'rce/url-encoded':     r'%73%79%73%74%65%6d',       # ← MT RCE で見た URL エンコード system
    'lfi/etc-passwd':      r'/etc/passwd',
    'path-traversal':      r'(?i)(\.\./|%2e%2e%2f|%252e%252e)',
    'wp-bruteforce':       r'POST /wp-login\.php',      # ← victor の WP 4.9.4 直撃
    'rainloop-known':      r'/rainloop/.*\?(version|admin)',  # rainloop 1.12.0 既知 CVE
    'pkexec-attempt':      r'pkexec',                   # PwnKit
}
```

**判断**: Mock で十分（既知パターン）。conf >= 0.85 なら自動 block 候補。

**反映**: 
- victor が nginx なら WAF ルールを `/etc/nginx/conf.d/` に追記
- Apache なら `mod_rewrite` で `RewriteCond %{QUERY_STRING}` ベースに変換（**追加レンダラ実装必要**）

**顧客対応文（Claude に書かせる例）**:
```
プロンプト: 「以下の WordPress への攻撃を検知し対応しました。
            一般顧客向けに 200 字以内で状況報告と謝罪を書いてください」
入力: 攻撃シグナル + 講じた対策
```

### 3.2 DDoS（白浜で頻出、別マニュアル `06_DDoS対応詳細.md` あり）

**観測対象**:
- `ss -s` / `netstat -an` の SYN_RECV カウント
- `tcpdump` の 1 秒あたりパケット数
- RTX1200/1210 の syslog（パケットドロップ）
- nginx の同一 IP からの大量リクエスト

**キャンプ知見の応用**:
- キャンプの `analyzer.py` に **時系列ウィンドウ集計**機能を足す
  - 「同一 IP から 60 秒で 1000 リクエスト」「SYN/ACK 比が極端」など
- 判断層で **Claude に「これは DDoS か正規バーストか」を文脈で判定させる**
  - 入力: 「アクセス元 IP 分布 + URL 分布 + UA 分布 + 直近 1 時間の比較」
  - 出力: `attack_type`, `recommended_action`, `customer_message_draft`

**反映パターン**（renderer に追加）:

```python
def render_ddos_mitigation(signal):
    if signal.attack_subtype == 'syn-flood':
        return [
            "iptables -A INPUT -p tcp --syn -m limit --limit 10/s -j ACCEPT",
            "iptables -A INPUT -p tcp --syn -j DROP",
            "sysctl -w net.ipv4.tcp_syncookies=1"
        ]
    elif signal.attack_subtype == 'http-flood':
        return [
            f"iptables -I INPUT -s {signal.source_ip} -j DROP",
            "nginx: limit_req_zone $binary_remote_addr zone=ddos:10m rate=10r/s;"
        ]
    elif signal.attack_subtype == 'dns-amplification':
        return [
            "named.conf: rate-limit { responses-per-second 5; window 5; };",
            "named.conf: allow-recursion { localnets; };"
        ]
```

**顧客対応の AI 活用**:
- 「お客様サイトにアクセスしづらいというお問い合わせ」→ Claude に状況サマリと**復旧見込み時間付きの返信文**を書かせる
- 既存の `13_DDoSアキネーター.html` と組み合わせて「**Claude が攻撃種別を絞り込み → アキネーターで人間が確認 → 反映**」のフロー

### 3.3 フィッシング・不審メール対応

**観測対象**:
- victor の `/var/log/maillog` (postfix)
- ユーザからの「不審メール届いたんだけど」電話通報（運営から来る）
- rainloop 経由で送信されたメールの履歴

**キャンプ知見の応用**:
これは Web より **Claude API の独壇場**。LLM の本領が活きる:

```python
def judge_phishing(mail_text: str, headers: dict) -> dict:
    """
    Claude に投げて以下を返してもらう:
    - is_phishing: bool
    - confidence: 0.0-1.0
    - indicators: ['偽装ドメイン', '緊急性煽り', 'リンク先不一致']
    - recommended_action: 'block' | 'quarantine' | 'warn' | 'pass'
    - customer_explanation: str  # ユーザに説明する文章
    """
```

**観測層で機械的に拾える特徴**（Claude に渡す前段）:
- `From:` と `Return-Path:` の不一致
- `Received:` の異常な経路
- リンクの表示文字列と URL の不一致
- 添付ファイル（.exe, .scr, .iso, マクロ付き Office）
- SPF / DKIM / DMARC の検証結果

**反映**:
- postfix `header_checks` / `body_checks` への追加
- 既に届いたメールの隔離（`postsuper -h`）
- ユーザへの注意喚起メール一斉配信（**雛形を Claude に書かせる**）

**顧客対応**:
- 通報した利用者への返信: 「ご通報ありがとうございます。当該メールは...」
- 全社向け注意喚起: 「**本日 X 時頃、当社を装ったフィッシングメールが...**」
- どちらも Claude に **「専門用語を使わず、何をすべきか明確に」** と指示して書かせる

### 3.4 DNS 攻撃（bravo の BIND allow-update 問題が直撃）

**前提**: `14_サーバ調査レポート` で判明済みの致命的設定:
```
allow-update { 10.0.0.0/8; };  # 10.x 全域から nsupdate でレコード書き換え可能
```

**観測対象**:
- `/var/log/named/` (もしくは syslog)
- 異常な `nsupdate` リクエスト
- AXFR を許可してない IP からの試行
- DNS amplification の踏み台にされてないか（`ANY` クエリ大量発生）

**シグネチャ**（analyzer 追加）:
```python
DNS_PATTERNS = {
    'dns/unauthorized-update':  r'client.*update.*denied',
    'dns/axfr-attempt':         r'transfer.*denied',
    'dns/amplification-bait':   r'query: .* IN ANY',
    'dns/unusual-record':       r'updating zone.*adding rdataset',  # nsupdate 成功
}
```

**判断**: Mock で即時。`updating zone` 検知 → **Critical** 即ブロック。

**反映**（**事前準備した named.conf パッチを当てる**）:
```
allow-update { none; };       # 即適用
allow-transfer { 10.1.130.1; };  # AXFR は secondary のみ
rate-limit { responses-per-second 10; window 5; };
```

→ ⚠️ **本番中に named.conf を直すのは怖い**。**事前に修正済み版を用意**しておき、当日は `cp` するだけにする。
   `17_本番前タスクリスト.md` に追加項目として書いておく。

### 3.5 SSH brute force / 既知 CVE 攻撃

**観測対象**:
- victor `/var/log/secure`
- bravo `/var/log/auth.log`
- `last`, `lastb`

**シグネチャ**:
```python
AUTH_PATTERNS = {
    'ssh/failed-burst':     r'Failed password.*from (\S+)',  # 集計して 60秒で >10 回
    'ssh/invalid-user':     r'Invalid user (\S+) from',
    'ssh/successful-after-fail': '失敗連発 → 同一IPから成功',  # 横断ロジック
    'sudo/unauthorized':    r'sudo:.*authentication failure',
    'pkexec/exploit':       r'pkexec.*GCONV_PATH',  # PwnKit シグネチャ
}
```

**判断層の見せ所**:
- 「**ログイン成功した IP は本当に正規ユーザか**」を Claude に判定させる
- 入力: 直近のログイン履歴 + その IP の過去アクセス + ユーザの活動時間帯
- 「obuchi が深夜 01:40 に 10.1.129.10 からログイン」みたいな兆候を Claude にプロンプトで投げる
  - キャンプの IDOR の例（教材 p.30）と同じ構造: **「リクエスト自体は正常。誰がやったかが問題」**

**反映**:
```bash
# 即時遮断
iptables -I INPUT -s {ip} -j DROP

# 中長期対策
# /etc/ssh/sshd_config:
# PasswordAuthentication no  ← 顧客承認必要
# PermitRootLogin no
# AllowUsers manage admin1 admin2

# fail2ban 設定 (事前準備推奨)
```

### 3.6 ネットワーク機器（RTX1200/1210, Cisco ASA, C3560-CX, Buffalo AP）

**ここはキャンプ知見の直接適用は難しい** が、**4 層思想は活きる**:

| 層 | NW 機器対応 |
|---|---|
| 観測 | RTX/ASA の syslog をリモートで集約（**事前に syslog forwarding 設定**） |
| 判断 | 「コンフィグに不正な変更があるか」を Claude に diff 判定させる |
| 検証 | 機器コマンド allowlist（`reload`, `erase startup`, `no enable` などは絶対禁止） |
| 提案 | コマンド生成は人間が二重確認してから流す |

**Claude にやらせる典型タスク**:
- 「RTX1200 のこのアクセスログ、攻撃か正規か判定して」
- 「Cisco ASA のこの ACL 設定、何を許可してる？日本語で」
- 「ヤマハ RTX のコマンドで `192.0.2.0/24` を遮断するには？」 ← 一次情報の代わり

**反映の鉄則**: NW 機器のコマンドは **絶対に AI 生成のまま流さない**。人間が一字一句確認。
   `enable secret` を不用意に変更すると **自分が締め出される** 事故が起こる（白浜過去事例）。

### 3.7 内部不正・横展開・ウェブシェル

**最も Claude が効く領域**（パターンマッチが効きにくい）。

**観測対象**:
- ファイルシステム異常（`find / -mtime -1 -type f`）
- 不審なプロセス（`ps auxf` の親子関係）
- `obuchi` ホームの 777（既知の踏み台候補）に新規ファイル
- victor の `wp-content/uploads/` に PHP ファイル（ウェブシェル）

**Claude プロンプト例**:

以下は victor の /tmp と /var/tmp の最近変更されたファイル一覧です。
怪しいものを優先順位付けして指摘してください。

入力: ls -lat /tmp /var/tmp の出力
出力: 怪しいファイル順 + 理由 + 推奨初動

→ これは「ノイズだらけのログから本物の脅威を絞り込む」というキャンプ Day2 のテーマと完全一致。

## 4. Claude / Mock / 観測層の使い分け（最重要）

```
即時性が必要 ─────────────────────→ 賢さが必要
   観測層        Mock         Claude
  (即時)       (即時)      (2-5秒)
   ↓           ↓             ↓
パターン       事前定義      文脈・自然言語・
検出のみ       ルール適用     未知攻撃判定
                            報告書生成
```

**判断ルール**:
1. **事前に既知のもの**（rainloop CVE, pkexec 等）→ Mock で即決、Claude を待たない
2. **ノイズの多いログを整理したい** → Claude で集約（17件→5件にする例）
3. **顧客向け文章 / 報告書 / 説明** → Claude 必須（人間が書くより速くて品質も安定）
4. **「これは攻撃か？」の判断が割れる** → Claude で文脈読み（IDOR / 内部不正 / フィッシング）
5. **NW 機器コマンド生成** → Claude に書かせて **必ず人間レビュー**（実機適用は手動）

**競技中の Claude の呼び方**:
```python
# パターン A: 手動トリガー（人間が「整理して」と言ったとき）
docker compose exec -w /app/ai-agent ai-agent python -c "
from agent import analyzer
from agent.backends.claude_backend import ClaudeBackend
signals = analyzer.run('/logs')
print(ClaudeBackend().explain_to_operator_ja(signals, []))
" > /tmp/situation_summary.txt

# パターン B: 自動定期実行（5 分に 1 回ログを集約）
# cron で回すか、シェルで while sleep 300 ループ
```

## 5. 検証層（Validator）の重要性 — 競技で事故を防ぐ最終防衛線

キャンプ Day2 §10.4 で「壊れた提案を全件拒否」のデモを見た。
**白浜では検証層の重要度がさらに上がる**。理由:

| 競技状況 | 起こりうる事故 | Validator が防ぐ |
|---|---|---|
| 焦って AI 提案を即実行 | `iptables -F` で全 ACL 飛ぶ | コマンド allowlist で `-F` 禁止 |
| Claude が幻覚で偽コマンド生成 | `systemctl stop sshd` で締め出し | 危険コマンドリストで拒否 |
| 顧客承認前に変更適用 | 評価で減点（プロセス遵守違反） | dry-run モード固定、適用は別フラグ |
| 大量提案を一気に流して環境破壊 | 全件 reject の方がマシ | all-or-nothing 仕様 |

**白浜用 validator の追加チェック**:
```python
DANGER_COMMANDS = [
    'iptables -F', 'iptables --flush',
    'rm -rf /', 'rm -rf /etc',
    'systemctl stop sshd', 'systemctl disable sshd',
    'shutdown', 'reboot', 'halt',
    'erase startup-config', 'reload',  # Cisco
    'cold start',  # YAMAHA RTX 工場出荷リセット
    'dd if=/dev/zero', 'mkfs',
    'DROP DATABASE', 'DROP TABLE',
    'allow-update { any', 'allow-transfer { any',
]

def validate_command(cmd: str):
    for danger in DANGER_COMMANDS:
        if danger in cmd:
            raise ValidationError(f"危険コマンド検出: {danger}")
```

## 9. アンチパターン（やってはいけないこと）

### 9.1 AI を盲信してそのまま実機反映

> Day2 §9.2.3 で見たように Claude は **同じ入力でも違う提案を出す**。  
> 1 回の出力を絶対視しない。複数回呼んで安定性を確認するか、人間が最終判断。

### 9.2 観測層を Claude に丸投げ

> 観測（パターン検出）は **analyzer.py のルールベース**で十分。  
> Claude を全ログに対して呼ぶと API レート制限・コスト・遅延で破綻する。  
> Claude は **「絞り込んだシグナルの判断」と「文章生成」** に集中させる。

### 9.3 検証層を抜くショートカット

> 「検証はあとで、まず動かす」は **競技で死ぬ**。  
> 検証層がない状態で AI 出力を流すと、`iptables -F` 一発で全 ACL 飛んで自分が締め出される。

### 9.4 顧客対応を AI 任せにして人間が読まない

> Claude の出力を **未読のまま顧客に送る**のは絶対 NG。  
> 評価軸の「顧客対応」は **人間が責任を持って一読・修正してから送る**。  
> AI は下書き生成までで、最終承認は人間。

### 9.5 NW 機器コマンドを AI 生成のまま流す

> RTX1200 の `cold start` や Cisco の `erase startup` を AI が出したら **環境ごと飛ぶ**。  
> NW 機器コマンドは **必ずベンダーマニュアル一次情報で確認**してから流す。  
> Claude は「マニュアルの代わり」ではなく「マニュアルを読む補助」に徹する。

### 9.6 競技中に新規にコード書く

> パイプラインや analyzer のルールを **競技中に書こうとしない**。  
> 事前準備で完成させ、当日は **実行と人間判断のみ**に集中する。

## 10. キャンプ Day1/Day2 の各シーンが白浜のどこに対応するか

| キャンプの体験 | 白浜での対応場面 |
|---|---|
| `<script>` が WAF で 403 になる | 既知パターンは Mock + 事前ルールで即遮断 |
| `<img onerror=>` が WAF を素通り | 未知バリアントは Claude で意味判定→新ルール追加 |
| MT RCE の URL エンコード突破 | victor の rainloop / WP に同種攻撃が来る前提で事前シグネチャ準備 |
| IDOR は WAF で止まらない | 認可系・内部不正は **アプリ/サーバ側** で対応必要と Claude が advisory 出してくれる |
| Claude が同じ入力で違う出力（§9.2.3） | 検証層 + 人間承認で吸収 |
| Swallow が `match_type='payload'` を出して validator が弾く | Claude でも起こりうる。validator が最後の砦 |
| `explain_to_operator_ja()` の SOC 報告文 | 経営層報告・顧客向け説明の **下書き生成器**として常用 |
| 「全部が WAF ルールになるわけではない」(p.40) | 「全部が技術対応で済むわけではない」(顧客対応・プロセス) と同型 |
| 「クリティカルだけ防ぐ」(p.25) | 競技時間内に全脆弱性は直せない。**致命的悪用だけ抑える**割り切り |
